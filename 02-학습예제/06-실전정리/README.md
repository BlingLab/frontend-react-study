## 실행 명령

```bash
pnpm run 예제:전체
```

실전 정리 단계에서는 특정 새 예제보다 전체 예제를 다시 훑으면서 설명하는 연습을 합니다.

## 확인할 코드

- `src/App.tsx`
- `src/examples/`

## 직접 해볼 것

- 기초부터 심화까지 예제를 하나씩 열어봅니다.
- 각 예제가 어떤 학습문서와 연결되는지 설명합니다.
- 예제 하나를 고르고 파일을 보지 않고 다시 작성합니다.
- `pnpm build`로 빌드가 통과하는지 확인합니다.
- 예제 하나에 대해 Given/When/Then 테스트 시나리오를 적어 봅니다.
- 버그를 일부러 만들고 render/event/Effect/API/CSS 중 어디 문제인지 분리해 봅니다.
- 변경 내용을 PR 설명 형식으로 정리해 봅니다.
- 키보드만으로 예제의 핵심 기능을 완료할 수 있는지 확인합니다.
- input에 label이 빠진 곳을 찾아 수정하고 `getByLabelText`로 테스트가 통과하는지 확인합니다.
- 종합 복습 시나리오의 게시글 검색 기능을 직접 처음부터 끝까지 구현해 봅니다.

## 예제 리뷰 체크리스트

| 항목 | 확인 질문 |
| --- | --- |
| 실행 | README의 명령만 보고 예제를 열 수 있는가? |
| 상태 | loading, error, empty, success 중 필요한 상태가 빠지지 않았는가? |
| 입력 | input에 label이 있고 키보드로 submit할 수 있는가? |
| 실패 | 실패했을 때 사용자가 다시 시도하거나 복구할 수 있는가? |
| 구조 | 컴포넌트 이름만 보고 역할을 추측할 수 있는가? |
| 타입 | 잘못된 props나 상태 조합을 TypeScript가 막아주는가? |
| 설명 | 왜 state, Effect, reducer, memoization을 썼는지 말할 수 있는가? |
| 검증 | `pnpm build`가 통과하는가? |

## PR 설명 연습 템플릿

예제 하나를 수정한 뒤 아래 형식으로 정리합니다.

```md
## 변경 내용

- 어떤 예제를 바꿨는지
- 어떤 사용자 흐름이 좋아졌는지

## 선택 이유

- state/props/Effect/reducer 중 무엇을 왜 선택했는지
- 다른 선택지를 쓰지 않은 이유

## 확인 방법

- 실행한 명령
- 브라우저에서 확인한 상태
- 남은 제한 사항
```
