# 예제 실습 가이드

이 문서는 예제를 그냥 실행해보는 데서 끝내지 않고, 문서에서 배운 개념을 코드로 확인하고 다시 설명하기 위한 공통 가이드입니다.

예제 하나를 볼 때는 항상 같은 순서로 봅니다.

1. 관련 문서를 먼저 읽고 핵심 질문을 적습니다.
2. 예제를 실행해 화면에서 어떤 상태가 보이는지 확인합니다.
3. 코드에서 state, props, event handler, Effect, 계산값을 표시합니다.
4. 작은 수정을 하나 넣고 화면 변화와 타입 오류를 확인합니다.
5. 파일을 닫고 같은 구조를 짧게 다시 만들어봅니다.
6. 왜 이 구조가 맞는지 말로 설명합니다.

## 예제 하나를 끝내는 기준

예제를 끝냈다고 보려면 아래 네 가지를 통과합니다.

| 기준 | 확인 방법 |
| --- | --- |
| 화면 이해 | 어떤 사용자 행동이 어떤 UI 상태를 만드는지 설명합니다. |
| 코드 이해 | state, props, handler, Effect, 계산값을 구분합니다. |
| 수정 경험 | 작은 요구사항 하나를 직접 바꾸고 타입 오류나 화면 변화를 확인합니다. |
| 재구현 | 파일을 닫고 같은 흐름을 10분 안에 다시 작성합니다. |

## 단계별 수정 과제

### 기초

- learner 데이터를 하나 추가하고 카드가 자동으로 늘어나는지 봅니다.
- skills가 빈 배열인 사용자를 추가해 empty UI를 확인합니다.
- `key`를 `index`로 바꿨을 때 왜 위험한지 말로 설명합니다.

### 초급

- Todo 삭제 버튼의 문구와 위치를 바꿔봅니다.
- 완료/미완료 필터를 추가하고, 필터링된 목록이 state인지 계산값인지 판단합니다.
- 입력값을 비운 채 제출했을 때 어디에서 막히는지 확인합니다.

### 중급

- fetch URL을 일부러 틀려 error UI와 retry 흐름을 봅니다.
- modal이 열렸을 때 focus와 닫기 동작을 확인합니다.
- 느린 목록 크기를 늘리고 `useDeferredValue`가 없는 버전과 비교합니다.

### 고급

- reducer action을 하나 추가하고 action 이름을 사용자 의도 중심으로 바꿉니다.
- transition 예제에서 `startTransition`을 제거해 반응성 차이를 확인합니다.
- optimistic 요청 실패 확률을 높이고 복구 UI가 충분한지 판단합니다.

### 심화

- memoization을 제거한 뒤 체감 차이와 코드 복잡도를 비교합니다.
- 목록 크기를 키운 뒤 먼저 측정하고, 그 다음에 최적화합니다.
- discriminated union에서 상태 필드를 잘못 조합했을 때 타입이 막는지 확인합니다.

## 검증 명령

문서만 바꿨더라도 예제 연결이 깨지지 않았는지 빌드로 확인합니다.

```bash
pnpm build
```

빌드 결과를 브라우저에서 확인하려면 아래처럼 preview 명령을 사용합니다.

```bash
pnpm run 미리보기:기초
pnpm run 미리보기:초급
pnpm run 미리보기:중급
pnpm run 미리보기:고급
pnpm run 미리보기:심화
```
