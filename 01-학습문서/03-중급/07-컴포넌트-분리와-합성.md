# 컴포넌트 분리와 합성

컴포넌트를 나눈다는 것은 파일을 많이 만드는 일이 아닙니다. 책임을 나누는 일입니다.

좋은 분리는 다음 질문에 답합니다.

- 이 컴포넌트는 무엇을 보여주는가?
- 이 컴포넌트는 어떤 state를 직접 관리해야 하는가?
- 이 컴포넌트가 몰라도 되는 세부사항은 무엇인가?
- props 이름만 보고 사용법을 이해할 수 있는가?

## 나누기 좋은 신호

아래 신호가 보이면 컴포넌트 분리를 고려합니다.

- 한 컴포넌트에서 데이터 요청, form, 목록, item UI, empty UI를 모두 처리한다.
- JSX가 길어서 조건부 렌더링의 의도가 잘 보이지 않는다.
- 같은 UI 조각이 여러 곳에서 반복된다.
- 특정 영역만 테스트하거나 확인하기 어렵다.
- props/state의 일부만 사용하는 영역이 뚜렷하다.

## 예제: 요청 상태 View 분리하기

처음에는 한 컴포넌트 안에 모두 있어도 됩니다.

```tsx
function PostsPage() {
  const { state, refetch } = usePosts();

  return (
    <section>
      <h1>게시글</h1>
      {state.status === "loading" && <p>불러오는 중...</p>}
      {state.status === "error" && <button onClick={refetch}>다시 시도</button>}
      {state.status === "empty" && <p>게시글이 없습니다.</p>}
      {state.status === "success" && <PostList posts={state.posts} />}
    </section>
  );
}
```

상태별 UI가 늘어나면 요청 상태를 그리는 컴포넌트로 분리할 수 있습니다.

```tsx
function PostsPage() {
  const { state, refetch } = usePosts();

  return (
    <section>
      <h1>게시글</h1>
      <PostsRequestView state={state} onRetry={refetch} />
    </section>
  );
}
```

```tsx
function PostsRequestView({
  state,
  onRetry,
}: {
  state: RequestState;
  onRetry: () => void;
}) {
  if (state.status === "loading") {
    return <p>불러오는 중...</p>;
  }

  if (state.status === "error") {
    return (
      <div role="alert">
        <p>{state.message}</p>
        <button onClick={onRetry}>다시 시도</button>
      </div>
    );
  }

  if (state.status === "empty") {
    return <p>게시글이 없습니다.</p>;
  }

  return <PostList posts={state.posts} />;
}
```

이렇게 분리하면 `PostsPage`는 큰 구조를 보여주고, `PostsRequestView`는 요청 상태별 UI를 책임집니다.

## props drilling과 composition

props를 여러 단계로 계속 전달하면 중간 컴포넌트가 실제로 쓰지 않는 props를 알게 됩니다.

```tsx
function Page() {
  return <Layout user={user} notifications={notifications} />;
}

function Layout({ user, notifications }: Props) {
  return <Header user={user} notifications={notifications} />;
}

function Header({ user, notifications }: Props) {
  return <UserMenu user={user} notifications={notifications} />;
}
```

이럴 때 context를 바로 쓰기 전에 composition을 고려합니다.

```tsx
function Page() {
  return (
    <Layout
      header={<UserMenu user={user} notifications={notifications} />}
      main={<Dashboard />}
    />
  );
}

function Layout({ header, main }: { header: ReactNode; main: ReactNode }) {
  return (
    <>
      <header>{header}</header>
      <main>{main}</main>
    </>
  );
}
```

`Layout`은 user와 notifications를 몰라도 됩니다. 필요한 완성된 조각을 받아 배치만 합니다.

## children으로 합성하기

공통 테두리, 제목, 버튼 영역을 가진 UI는 `children`으로 합성하기 좋습니다.

```tsx
function Panel({
  title,
  actions,
  children,
}: {
  title: string;
  actions?: ReactNode;
  children: ReactNode;
}) {
  return (
    <section className="panel">
      <header>
        <h2>{title}</h2>
        {actions}
      </header>
      <div>{children}</div>
    </section>
  );
}
```

사용하는 쪽은 화면의 의미를 그대로 유지합니다.

```tsx
<Panel title="게시글" actions={<button onClick={refetch}>새로고침</button>}>
  <PostsRequestView state={state} onRetry={refetch} />
</Panel>
```

## container와 presentational을 너무 엄격히 나누지 않기

예전에는 "container component"와 "presentational component"를 강하게 나누는 방식이 많이 쓰였습니다. 지금도 유용한 구분이지만 규칙처럼 강제할 필요는 없습니다.

현실적인 기준은 이렇습니다.

- URL, 데이터 요청, 권한 체크는 page나 상위 컴포넌트에 둡니다.
- 순수하게 props를 받아 그리는 UI는 하위 컴포넌트로 둡니다.
- 작은 interaction은 해당 컴포넌트 안에 둬도 됩니다.
- 여러 곳에서 같은 interaction 규칙이 반복되면 custom Hook으로 뽑습니다.

## 너무 잘게 나누는 것도 비용이다

컴포넌트를 너무 잘게 나누면 오히려 흐름을 따라가기 어렵습니다. 아래처럼 한 줄짜리 컴포넌트가 파일마다 흩어지면 읽는 비용이 커질 수 있습니다.

```tsx
function ProductTitle({ title }: { title: string }) {
  return <h3>{title}</h3>;
}
```

이런 분리는 재사용되거나 의미 있는 책임이 생길 때까지 기다려도 됩니다.

## 좋은 분리의 기준

| 기준 | 설명 |
| --- | --- |
| 이름 | 컴포넌트 이름이 역할을 설명하는가? |
| props | 필요한 값만 받는가? |
| 책임 | 데이터 요청과 표시 책임이 뒤섞이지 않았는가? |
| 재사용 | 같은 패턴이 반복되는가? |
| 독립성 | 분리된 컴포넌트를 따로 읽어도 이해되는가? |

## 읽으면서 생각할 질문

- 이 컴포넌트의 주된 책임은 무엇인가?
- 중간 컴포넌트가 쓰지 않는 props를 전달만 하고 있지는 않은가?
- context를 쓰기 전에 composition으로 해결할 수 있는가?
- 하위 컴포넌트가 router나 fetch 세부사항을 몰라도 되는가?
- 분리한 뒤 호출부가 더 읽기 쉬워졌는가?
