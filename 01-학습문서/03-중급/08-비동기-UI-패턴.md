# 비동기 UI 패턴

비동기 UI는 "기다렸다가 성공하면 보여준다"보다 넓게 봐야 합니다. 사용자는 요청이 느리거나 실패하거나 중복 클릭되거나 결과가 비어 있는 상황을 모두 경험합니다.

중급 단계에서는 API 호출 자체보다 요청 전후의 화면 상태와 사용자 행동을 설계하는 능력이 중요합니다.

## 기본 상태 흐름

가장 기본적인 흐름은 다음과 같습니다.

```
idle → loading → success
idle → loading → empty
idle → loading → error
error → loading → success  (재시도)
```

이 흐름이 코드와 UI에 드러나야 합니다.

```tsx
type AsyncState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "error"; message: string }
  | { status: "empty" }
  | { status: "success"; data: T };
```

## 버튼 disabled는 비동기 상태와 연결한다

저장 요청 중에 저장 버튼을 계속 누를 수 있으면 같은 요청이 여러 번 발생할 수 있습니다.

```tsx
function SaveButton({ status, onSave }: Props) {
  const isSaving = status === "loading";

  return (
    <button type="button" disabled={isSaving} onClick={onSave}>
      {isSaving ? "저장 중..." : "저장"}
    </button>
  );
}
```

disabled는 단순한 스타일이 아닙니다. 중복 요청과 이상한 상태 전환을 막는 기능입니다.

## 이전 데이터 유지하기

새 검색어로 다시 요청할 때 기존 데이터를 바로 지울지 유지할지 선택해야 합니다.

**기존 데이터를 지우는 방식**

```tsx
setState({ status: "loading" });
```

장점은 단순합니다. 단점은 화면이 매번 비었다가 다시 채워져 깜빡일 수 있습니다.

**기존 데이터를 유지하는 방식**

```tsx
type PostsState =
  | { status: "loading"; previousData?: Post[] }
  | { status: "success"; data: Post[] }
  | { status: "error"; message: string; previousData?: Post[] };
```

이 방식은 UX가 부드럽지만 상태가 복잡해집니다.

```tsx
if (state.status === "loading" && state.previousData) {
  return (
    <>
      <p>새 결과를 불러오는 중...</p>
      <PostList posts={state.previousData} dimmed />
    </>
  );
}
```

둘 중 항상 정답은 없습니다. 검색 결과처럼 바뀌는 맥락이 중요하면 지우는 편이 낫고, 새로고침처럼 같은 데이터를 갱신하는 경우에는 이전 데이터를 유지하는 편이 자연스럽습니다.

## stale response 방지

사용자가 검색어를 빠르게 바꾸면 요청이 여러 개 동시에 진행될 수 있습니다.

```
요청 A: keyword=react   → 늦게 도착
요청 B: keyword=router  → 먼저 도착
```

요청 A가 나중에 도착해서 B의 결과를 덮어쓰면 화면은 오래된 결과를 보여줍니다. Effect cleanup에서 ignore 플래그를 쓰거나 `AbortController`를 사용해 방지합니다.

```tsx
useEffect(() => {
  let ignore = false;

  async function run() {
    const result = await search(keyword);

    if (!ignore) {
      setResult(result);
    }
  }

  run();

  return () => {
    ignore = true;
  };
}, [keyword]);
```

## retry UI

에러 메시지만 보여주고 끝내면 사용자는 다음 행동을 알 수 없습니다. 네트워크 오류나 일시적인 서버 오류라면 다시 시도 버튼이 필요합니다.

```tsx
function ErrorState({
  message,
  onRetry,
}: {
  message: string;
  onRetry: () => void;
}) {
  return (
    <div role="alert">
      <p>{message}</p>
      <button type="button" onClick={onRetry}>
        다시 시도
      </button>
    </div>
  );
}
```

에러 UI에는 보통 아래 정보가 들어갑니다.

- 무엇이 실패했는지
- 다시 시도할 수 있는지
- 입력값을 바꿔야 하는지
- 권한이나 로그인 문제가 있는지

## optimistic update

사용자 행동을 즉시 화면에 반영하고, 서버 요청이 실패하면 되돌리는 패턴입니다. 좋아요, 체크박스, 간단한 토글에서 자주 씁니다.

```tsx
async function handleLikeClick(postId: string) {
  // 즉시 반영
  setLikedIds((ids) => [...ids, postId]);

  try {
    await likePost(postId);
    // 성공 — 이미 반영되어 있음
  } catch {
    // 실패 — 되돌리기
    setLikedIds((ids) => ids.filter((id) => id !== postId));
    showToast("좋아요 저장에 실패했습니다.");
  }
}
```

optimistic update는 빠른 UI를 만들지만 실패 복구를 반드시 생각해야 합니다. 결제, 권한 변경, 삭제처럼 되돌리기 어려운 작업에는 신중해야 합니다.

## 병렬 요청

여러 API를 동시에 요청할 때는 `Promise.all`을 씁니다.

```tsx
useEffect(() => {
  let ignore = false;

  async function loadAll() {
    setState({ status: "loading" });

    try {
      const [user, posts] = await Promise.all([
        fetchUser(userId),
        fetchUserPosts(userId),
      ]);

      if (!ignore) {
        setState({ status: "success", data: { user, posts } });
      }
    } catch (error) {
      if (!ignore) {
        setState({ status: "error", message: getErrorMessage(error) });
      }
    }
  }

  loadAll();
  return () => { ignore = true; };
}, [userId]);
```

`Promise.allSettled`는 하나가 실패해도 나머지 결과를 받고 싶을 때 씁니다.

```tsx
const [userResult, postsResult] = await Promise.allSettled([
  fetchUser(userId),
  fetchUserPosts(userId),
]);

// 각 결과를 독립적으로 처리
const user = userResult.status === "fulfilled" ? userResult.value : null;
const posts = postsResult.status === "fulfilled" ? postsResult.value : [];
```

## 순차 요청

첫 번째 요청의 결과로 두 번째 요청을 해야 할 때입니다.

```tsx
useEffect(() => {
  let ignore = false;

  async function load() {
    setState({ status: "loading" });

    try {
      const user = await fetchUser(userId);
      const posts = await fetchPostsByCategory(user.preferredCategory);

      if (!ignore) {
        setState({ status: "success", data: { user, posts } });
      }
    } catch (error) {
      if (!ignore) {
        setState({ status: "error", message: getErrorMessage(error) });
      }
    }
  }

  load();
  return () => { ignore = true; };
}, [userId]);
```

순차 요청은 첫 번째 결과가 없으면 두 번째 요청을 시작할 수 없으므로 전체 로딩 시간이 길어집니다. 병렬 처리가 가능한지 먼저 확인합니다.

## skeleton, spinner, text

loading UI도 상황에 따라 다릅니다.

| UI | 어울리는 경우 |
| --- | --- |
| text ("불러오는 중...") | 작은 영역, 짧은 요청 |
| spinner | 동작 중임을 간단히 보여줄 때 |
| skeleton | 화면 구조가 있고 콘텐츠만 늦게 올 때 |
| previous data dim | 기존 데이터를 유지하며 갱신할 때 |

항상 화려한 skeleton이 좋은 것은 아닙니다. 작은 버튼 저장에는 "저장 중..." 텍스트가 더 명확할 수 있습니다.

skeleton은 레이아웃이 안정적이고 콘텐츠가 비교적 오래 걸릴 때 효과적입니다. 200ms 이내로 완료되는 요청에 skeleton을 쓰면 오히려 깜빡임이 더 눈에 띌 수 있습니다.

## 비동기 작업을 컴포넌트에서 어디까지 알게 할까

작은 기능에서는 컴포넌트 안에서 fetch를 직접 호출해도 됩니다. 하지만 컴포넌트가 URL, method, response parsing, error mapping을 모두 알게 되면 화면 코드가 무거워집니다.

```tsx
// API 함수 분리
async function fetchPosts(): Promise<Post[]> {
  const response = await fetch("/api/posts");

  if (!response.ok) {
    throw new Error("게시글을 불러오지 못했습니다.");
  }

  return (await response.json()) as Post[];
}

function getErrorMessage(error: unknown): string {
  if (error instanceof Error) return error.message;
  return "알 수 없는 오류가 발생했습니다.";
}
```

컴포넌트는 이 함수를 호출하고 UI 상태에 집중합니다.

```tsx
async function load() {
  setState({ status: "loading" });

  try {
    const posts = await fetchPosts();
    setState(
      posts.length === 0
        ? { status: "empty" }
        : { status: "success", data: posts },
    );
  } catch (error) {
    setState({ status: "error", message: getErrorMessage(error) });
  }
}
```

## 읽으면서 생각할 질문

- 요청 중 사용자가 다시 버튼을 누를 수 있어야 하는가?
- 실패했을 때 복구 행동이 있는가?
- 빈 결과와 실패를 구분하고 있는가?
- 이전 데이터를 유지하는 편이 자연스러운가, 지우는 편이 자연스러운가?
- 오래된 응답이 최신 응답을 덮어쓸 가능성은 없는가?
- optimistic update가 실패했을 때 되돌릴 수 있는가?
- 병렬로 처리할 수 있는 요청을 순차로 처리하고 있지는 않은가?
- loading UI가 요청 시간과 영역 크기에 맞게 선택되었는가?
