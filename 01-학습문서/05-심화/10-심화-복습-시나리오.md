# 심화 복습 시나리오

심화 단계의 개념을 실제 코드로 읽고 설명할 수 있는지 확인합니다. 각 시나리오에서 "이 코드는 무엇을 하는가, 왜 이렇게 구성됐는가, 어떤 트레이드오프가 있는가"를 말로 설명해봅니다.

## 시나리오 1. 렌더링 비용 판단하기

아래 컴포넌트를 읽고 성능 문제가 있는지 판단합니다.

```tsx
function ProductPage() {
  const [keyword, setKeyword] = useState("");
  const [products, setProducts] = useState<Product[]>([]);

  useEffect(() => {
    fetchProducts().then(setProducts);
  }, []);

  const filtered = products.filter((p) =>
    p.name.toLowerCase().includes(keyword.toLowerCase())
  );

  return (
    <>
      <input
        value={keyword}
        onChange={(e) => setKeyword(e.target.value)}
        placeholder="제품 검색"
      />
      <p>{filtered.length}개 결과</p>
      <ProductList products={filtered} />
    </>
  );
}
```

설명할 수 있어야 할 것:

- `keyword`가 바뀔 때마다 `products.filter`가 실행됩니다.
- 제품이 수십 개라면 이 계산은 가볍습니다. memoization이 필요 없을 수 있습니다.
- 제품이 수천 개이거나 필터 로직이 복잡하다면 먼저 Profiler로 측정합니다.
- 측정 결과 병목이 확인된 뒤에야 `useMemo`를 추가합니다.

```tsx
// 측정 후 병목이 확인됐을 때만
const filtered = useMemo(
  () => products.filter((p) => p.name.toLowerCase().includes(keyword.toLowerCase())),
  [products, keyword],
);
```

## 시나리오 2. memo가 깨지는 상황 파악하기

아래 코드에서 `ProductRow`가 `memo`로 감싸져 있는데도 매번 렌더링되는 이유를 설명합니다.

```tsx
const ProductRow = memo(function ProductRow({
  product,
  onSelect,
}: {
  product: Product;
  onSelect: (id: string) => void;
}) {
  return <button onClick={() => onSelect(product.id)}>{product.name}</button>;
});

function ProductList({ products }: { products: Product[] }) {
  const [selectedId, setSelectedId] = useState<string | null>(null);

  return (
    <ul>
      {products.map((product) => (
        <ProductRow
          key={product.id}
          product={product}
          onSelect={(id) => setSelectedId(id)}  // ← 매 렌더마다 새 함수
        />
      ))}
    </ul>
  );
}
```

설명할 수 있어야 할 것:

- `(id) => setSelectedId(id)`는 렌더링마다 새로 만들어집니다.
- `memo`는 props 참조를 비교하므로, 함수가 새로 만들어지면 "props가 바뀐 것"으로 봅니다.
- 해결책은 항상 `useCallback`이 아닙니다.

```tsx
// 방법 1: product에서 id를 꺼내는 책임을 자식에게 줌
onSelect={setSelectedId}  // 함수 자체를 전달, 람다 제거

// 방법 2: 자식이 id를 알고 있으면 handler를 단순화
function ProductRow({ product, onSelect }: ...) {
  return <button onClick={() => onSelect(product.id)}>{product.name}</button>;
}
// 부모에서 setSelectedId 직접 전달
<ProductRow product={product} onSelect={setSelectedId} />
```

`setSelectedId`는 React가 안정적인 참조를 보장하는 함수이므로 렌더링마다 바뀌지 않습니다.

## 시나리오 3. Suspense boundary 위치 결정하기

아래 두 구조 중 어떤 게 더 나은지 판단하고 이유를 설명합니다.

**구조 A: 한 번에 묶기**

```tsx
function DashboardPage() {
  return (
    <Suspense fallback={<DashboardSkeleton />}>
      <UserProfile />
      <ActivityFeed />
      <NotificationList />
    </Suspense>
  );
}
```

**구조 B: 각자 경계 나누기**

```tsx
function DashboardPage() {
  return (
    <div>
      <Suspense fallback={<ProfileSkeleton />}>
        <UserProfile />
      </Suspense>
      <Suspense fallback={<FeedSkeleton />}>
        <ActivityFeed />
      </Suspense>
      <Suspense fallback={<NotificationSkeleton />}>
        <NotificationList />
      </Suspense>
    </div>
  );
}
```

설명할 수 있어야 할 것:

- 구조 A: `UserProfile`만 느려도 세 영역이 모두 skeleton으로 바뀝니다.
- 구조 B: 각자 독립적으로 로딩됩니다. 하나가 느려도 다른 영역은 바로 보입니다.
- 항상 B가 좋은 것은 아닙니다. 데이터가 강하게 연결된 영역은 A처럼 묶는 것이 자연스러울 수 있습니다.
- 사용자가 "무엇을 기다리는가"를 기준으로 경계를 정합니다.

## 시나리오 4. Custom Hook 추출 판단하기

아래 컴포넌트에서 어떤 로직을 Custom Hook으로 추출할지 판단합니다.

```tsx
function PostSearch() {
  const [keyword, setKeyword] = useState("");
  const [results, setResults] = useState<Post[]>([]);
  const [status, setStatus] = useState<"idle" | "loading" | "error">("idle");

  useEffect(() => {
    if (!keyword.trim()) {
      setResults([]);
      setStatus("idle");
      return;
    }

    setStatus("loading");

    const controller = new AbortController();

    fetchPosts(keyword, { signal: controller.signal })
      .then((data) => {
        setResults(data);
        setStatus("idle");
      })
      .catch((err) => {
        if (err.name !== "AbortError") {
          setStatus("error");
        }
      });

    return () => controller.abort();
  }, [keyword]);

  return (
    <>
      <input value={keyword} onChange={(e) => setKeyword(e.target.value)} />
      {status === "loading" && <p>검색 중...</p>}
      {status === "error" && <p role="alert">검색에 실패했습니다.</p>}
      <PostList posts={results} />
    </>
  );
}
```

설명할 수 있어야 할 것:

- 검색 로직(상태 관리, Effect, cleanup)은 UI와 분리 가능합니다.
- 같은 로직이 다른 페이지에서도 필요하거나, 컴포넌트가 화면 계산과 상태 관리를 함께 해서 읽기 어렵다면 추출합니다.

```tsx
function usePostSearch(keyword: string) {
  const [results, setResults] = useState<Post[]>([]);
  const [status, setStatus] = useState<"idle" | "loading" | "error">("idle");

  useEffect(() => {
    if (!keyword.trim()) {
      setResults([]);
      setStatus("idle");
      return;
    }

    setStatus("loading");
    const controller = new AbortController();

    fetchPosts(keyword, { signal: controller.signal })
      .then((data) => {
        setResults(data);
        setStatus("idle");
      })
      .catch((err) => {
        if (err.name !== "AbortError") setStatus("error");
      });

    return () => controller.abort();
  }, [keyword]);

  return { results, status };
}

function PostSearch() {
  const [keyword, setKeyword] = useState("");
  const { results, status } = usePostSearch(keyword);

  return (
    <>
      <input value={keyword} onChange={(e) => setKeyword(e.target.value)} />
      {status === "loading" && <p>검색 중...</p>}
      {status === "error" && <p role="alert">검색에 실패했습니다.</p>}
      <PostList posts={results} />
    </>
  );
}
```

## 시나리오 5. 렌더링 범위 줄이기

아래 코드에서 불필요한 렌더링 범위를 파악하고 구조를 개선합니다.

```tsx
function ProductPage() {
  const [keyword, setKeyword] = useState("");
  const [isFilterOpen, setIsFilterOpen] = useState(false);

  return (
    <>
      <FilterPanel open={isFilterOpen} onClose={() => setIsFilterOpen(false)} />
      <button onClick={() => setIsFilterOpen(true)}>필터 열기</button>
      <SearchInput value={keyword} onChange={setKeyword} />
      <ExpensiveProductChart keyword={keyword} />
    </>
  );
}
```

설명할 수 있어야 할 것:

- `isFilterOpen`이 바뀌면 `ExpensiveProductChart`도 다시 계산될 수 있습니다.
- `ExpensiveProductChart`는 `isFilterOpen`과 무관합니다.
- state 위치와 컴포넌트 분리로 업데이트 범위를 줄일 수 있습니다.

```tsx
function FilterButton() {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <>
      <FilterPanel open={isOpen} onClose={() => setIsOpen(false)} />
      <button onClick={() => setIsOpen(true)}>필터 열기</button>
    </>
  );
}

function SearchSection() {
  const [keyword, setKeyword] = useState("");

  return (
    <>
      <SearchInput value={keyword} onChange={setKeyword} />
      <ExpensiveProductChart keyword={keyword} />
    </>
  );
}

function ProductPage() {
  return (
    <>
      <FilterButton />
      <SearchSection />
    </>
  );
}
```

`isFilterOpen`이 바뀌어도 `SearchSection`과 `ExpensiveProductChart`는 다시 계산되지 않습니다.

## 시나리오 6. TypeScript 패턴 읽기

아래 코드에서 타입 설계의 의도를 설명합니다.

```tsx
type SearchState =
  | { phase: "idle" }
  | { phase: "loading"; keyword: string }
  | { phase: "success"; keyword: string; results: Post[] }
  | { phase: "error"; keyword: string; message: string };

function useSearch(): {
  state: SearchState;
  search: (keyword: string) => void;
  reset: () => void;
} {
  const [state, setState] = useState<SearchState>({ phase: "idle" });

  function search(keyword: string) {
    setState({ phase: "loading", keyword });

    fetchPosts(keyword)
      .then((results) => setState({ phase: "success", keyword, results }))
      .catch((err) => setState({ phase: "error", keyword, message: err.message }));
  }

  function reset() {
    setState({ phase: "idle" });
  }

  return { state, search, reset };
}
```

설명할 수 있어야 할 것:

- `phase`로 좁힌 뒤 해당 단계에서만 있는 필드(`results`, `message`)에 접근할 수 있습니다.
- `loading` 상태에서 `results`에 접근하는 실수를 컴파일 단계에서 막습니다.
- Hook 반환 타입을 명시하면 사용하는 쪽에서 의도한 인터페이스만 보입니다.

## 시나리오 7. 순수한 렌더링인지 판단하기

아래 두 컴포넌트 중 문제가 있는 쪽을 찾고 이유를 설명합니다.

**A:**
```tsx
let renderCount = 0;

function DebugBadge({ label }: { label: string }) {
  renderCount++;

  return <span>{label} (render #{renderCount})</span>;
}
```

**B:**
```tsx
function DebugBadge({ label }: { label: string }) {
  const [renderCount, setRenderCount] = useState(0);

  useEffect(() => {
    setRenderCount((c) => c + 1);
  });

  return <span>{label} (render #{renderCount})</span>;
}
```

설명할 수 있어야 할 것:

- A는 module scope 변수를 렌더링 중에 바꿉니다. 여러 컴포넌트 인스턴스가 값을 공유하고, Strict Mode에서 두 번 호출되면 예상과 다른 결과가 나옵니다.
- B는 Effect를 써서 렌더링 후 카운트를 올리지만, 매번 추가 렌더링을 유발합니다.
- 렌더 횟수 추적은 React DevTools Profiler나 `useRef`로 처리하는 편이 낫습니다.

```tsx
function DebugBadge({ label }: { label: string }) {
  const renderCountRef = useRef(0);
  renderCountRef.current++;

  return <span>{label} (render #{renderCountRef.current})</span>;
}
```

`ref`를 바꾸는 것은 렌더링 외부 상태를 바꾸지 않으므로 순수성을 해치지 않습니다.

## 복습 순서

심화 단계 복습은 아래 순서로 하면 좋습니다.

1. 느리다고 느끼는 구체적인 사용자 행동을 정합니다.
2. Profiler로 어느 컴포넌트가 느린지 봅니다.
3. state 위치와 컴포넌트 분리로 업데이트 범위를 줄일 수 있는지 봅니다.
4. 비싼 계산이 있다면 `useMemo`를 검토합니다.
5. memoized 자식에게 매번 새 props가 가는지 봅니다.
6. 큰 목록이 있다면 pagination이나 virtualization을 먼저 검토합니다.
7. 렌더링 중 외부 상태를 바꾸는 곳이 없는지 확인합니다.
8. Custom Hook 추출이 컴포넌트 가독성을 높이는지 판단합니다.

## 생각 흐름으로 점검하기

1. **이 렌더링이 실제로 느린가?** → 체감이 아닌 Profiler 수치로 확인
2. **느린 계산인가, 많은 DOM node인가, 넓은 업데이트 범위인가?** → 원인 분류
3. **state 위치나 컴포넌트 분리로 해결되는가?** → 구조 먼저
4. **memoization이 실제로 도움이 되는가?** → 적용 후 재측정
5. **렌더링이 순수한가?** → 외부 변경 없는지 확인
6. **TypeScript 타입이 잘못된 상태 조합을 막는가?** → discriminated union 확인
