# 큰 목록과 렌더링 범위 줄이기

큰 목록은 React 성능 문제에서 가장 자주 만나는 사례입니다. 수천 개 item을 한 번에 필터링하고, 정렬하고, DOM으로 그리면 입력과 스크롤이 느려질 수 있습니다.

이때 `useMemo`만 떠올리면 부족합니다. 계산 비용, DOM node 수, 업데이트 범위를 따로 봐야 합니다.

## 큰 목록의 세 가지 비용

| 비용 | 예 | 먼저 볼 해결책 |
| --- | --- | --- |
| 계산 비용 | filter, sort, groupBy가 비쌈 | `useMemo`, 알고리즘 개선, 서버 검색 |
| 렌더링 비용 | item 컴포넌트가 무거움 | 컴포넌트 분리, `memo`, props 안정화 |
| DOM 비용 | node가 너무 많음 | pagination, infinite scroll, virtualization |

어떤 비용이 병목인지에 따라 해결책이 다릅니다.

## pagination부터 생각하기

모든 데이터를 한 화면에 보여줄 필요가 없다면 pagination이 가장 단순합니다.

```tsx
const pageItems = items.slice(page * pageSize, (page + 1) * pageSize);
```

pagination은 DOM node 수를 줄이고, 사용자가 한 번에 처리해야 할 정보량도 줄입니다.

## infinite scroll은 UX 요구가 있을 때

infinite scroll은 자연스럽지만 구현 복잡도가 있습니다.

- 현재까지 불러온 데이터와 다음 page 상태를 관리해야 합니다.
- loading, error, retry 위치가 목록 하단으로 이동합니다.
- 사용자가 footer에 도달하기 어려울 수 있습니다.
- URL과 스크롤 복원 전략이 필요할 수 있습니다.

단순 관리자 화면이나 검색 결과는 pagination이 더 명확한 경우가 많습니다.

## virtualization

virtualization은 화면에 보이는 item 근처만 DOM으로 렌더링하는 방식입니다. 목록이 매우 크고 스크롤 UX가 중요할 때 고려합니다.

예를 들어 10,000개 item이 있어도 실제 DOM에는 화면에 보이는 수십 개만 렌더링합니다.

```tsx
// 개념 예시
const visibleItems = items.slice(startIndex, endIndex);
```

실전에서는 직접 구현보다 검증된 라이브러리를 쓰는 편이 안전합니다. item 높이가 가변인지, keyboard navigation이 필요한지, sticky header가 있는지에 따라 난도가 올라갑니다.

## state 위치로 렌더링 범위 줄이기

큰 목록과 무관한 state가 목록 부모에 있으면 목록이 자주 다시 계산될 수 있습니다.

```tsx
function Page() {
  const [keyword, setKeyword] = useState("");
  const [isHelpOpen, setIsHelpOpen] = useState(false);

  return (
    <>
      <HelpModal open={isHelpOpen} />
      <SearchInput value={keyword} onChange={setKeyword} />
      <HugeList keyword={keyword} />
    </>
  );
}
```

`isHelpOpen`이 바뀔 때 `HugeList`가 다시 계산될 필요가 없다면 state 위치나 컴포넌트 구조를 조정합니다.

```tsx
function Page() {
  return (
    <>
      <HelpButton />
      <SearchableHugeList />
    </>
  );
}
```

최적화는 Hook 하나보다 구조 변경이 더 효과적일 때가 많습니다.

## deferred value와 큰 목록

검색 input은 즉시 반응하고, 목록 업데이트는 조금 늦게 따라와도 된다면 `useDeferredValue`를 고려합니다.

```tsx
const deferredKeyword = useDeferredValue(keyword);

return (
  <>
    <input value={keyword} onChange={(event) => setKeyword(event.target.value)} />
    <HugeList keyword={deferredKeyword} />
  </>
);
```

이 방식은 API 요청 횟수를 줄이는 debounce가 아닙니다. 느린 렌더링 때문에 입력이 막히는 문제를 완화하는 도구입니다.

## 읽으면서 생각할 질문

- 느린 원인이 계산, 렌더링, DOM node 수 중 무엇인가?
- 한 화면에 모든 item을 보여줘야 하는가?
- pagination으로 충분한 문제를 virtualization으로 과하게 풀고 있지는 않은가?
- 목록과 무관한 state가 목록 부모에 있어 렌더링 범위를 넓히고 있지는 않은가?
- 사용자가 입력하는 동안 목록이 늦게 따라와도 괜찮은가?
