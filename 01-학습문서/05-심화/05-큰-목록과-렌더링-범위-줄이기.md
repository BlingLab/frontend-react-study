# 큰 목록과 렌더링 범위 줄이기

큰 목록은 React 성능 문제에서 가장 자주 만나는 사례입니다. 수천 개 item을 한 번에 필터링하고, 정렬하고, DOM으로 그리면 입력과 스크롤이 느려질 수 있습니다.

이때 `useMemo`만 떠올리면 부족합니다. 계산 비용, DOM node 수, 업데이트 범위를 따로 봐야 합니다.

## 큰 목록의 세 가지 비용

| 비용 | 예 | 먼저 볼 해결책 |
| --- | --- | --- |
| 계산 비용 | filter, sort, groupBy가 비쌈 | `useMemo`, 알고리즘 개선, 서버 검색 |
| 렌더링 비용 | item 컴포넌트가 무거움 | 컴포넌트 분리, `memo`, props 안정화 |
| DOM 비용 | node가 너무 많음 | pagination, infinite scroll, virtualization |

어떤 비용이 병목인지에 따라 해결책이 다릅니다. Profiler로 먼저 원인을 파악합니다.

## pagination부터 생각하기

모든 데이터를 한 화면에 보여줄 필요가 없다면 pagination이 가장 단순합니다.

```tsx
const PAGE_SIZE = 20;

function PaginatedList({ items }: { items: Item[] }) {
  const [page, setPage] = useState(0);
  const totalPages = Math.ceil(items.length / PAGE_SIZE);
  const pageItems = items.slice(page * PAGE_SIZE, (page + 1) * PAGE_SIZE);

  return (
    <>
      <ul>
        {pageItems.map((item) => (
          <li key={item.id}>{item.name}</li>
        ))}
      </ul>
      <div>
        <button disabled={page === 0} onClick={() => setPage((p) => p - 1)}>
          이전
        </button>
        <span>{page + 1} / {totalPages}</span>
        <button
          disabled={page >= totalPages - 1}
          onClick={() => setPage((p) => p + 1)}
        >
          다음
        </button>
      </div>
    </>
  );
}
```

pagination은 DOM node 수를 줄이고, 사용자가 한 번에 처리해야 할 정보량도 줄입니다. 관리자 화면, 검색 결과, 데이터 테이블에서 많이 씁니다.

## infinite scroll은 UX 요구가 있을 때

infinite scroll은 자연스럽지만 구현 복잡도가 있습니다.

- 현재까지 불러온 데이터와 다음 page 상태를 관리해야 합니다.
- loading, error, retry 위치가 목록 하단으로 이동합니다.
- 사용자가 footer에 도달하기 어려울 수 있습니다.
- URL과 스크롤 복원 전략이 필요할 수 있습니다.

단순 관리자 화면이나 검색 결과는 pagination이 더 명확한 경우가 많습니다.

**IntersectionObserver로 마지막 항목 감지:**

```tsx
function useInfiniteScroll(onLoadMore: () => void) {
  const sentinelRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    const observer = new IntersectionObserver(
      (entries) => {
        if (entries[0].isIntersecting) {
          onLoadMore();
        }
      },
      { rootMargin: "100px" }
    );

    if (sentinelRef.current) {
      observer.observe(sentinelRef.current);
    }

    return () => observer.disconnect();
  }, [onLoadMore]);

  return sentinelRef;
}

function InfiniteList({ items, onLoadMore, hasMore }: Props) {
  const sentinelRef = useInfiniteScroll(onLoadMore);

  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>{item.name}</li>
      ))}
      {hasMore && <div ref={sentinelRef} style={{ height: 1 }} />}
    </ul>
  );
}
```

## virtualization

virtualization은 화면에 보이는 item 근처만 DOM으로 렌더링하는 방식입니다. 목록이 매우 크고 스크롤 UX가 중요할 때 고려합니다.

예를 들어 10,000개 item이 있어도 실제 DOM에는 화면에 보이는 수십 개만 렌더링합니다. 스크롤하면 viewport 밖의 item은 DOM에서 제거되고 새 item이 추가됩니다.

**virtualization이 필요한 신호:**
- 목록이 1,000개 이상이고 스크롤 UX가 중요합니다.
- 각 item이 이미지, 차트 같은 무거운 컴포넌트입니다.
- 스크롤할 때 FPS가 60 아래로 떨어집니다.

**자주 쓰는 라이브러리:**

| 라이브러리 | 특징 |
| --- | --- |
| `@tanstack/react-virtual` | 경량, 다양한 레이아웃 지원, 가변 행 높이 |
| `react-window` | 안정적, 고정 크기 최적화 |
| `react-virtuoso` | 자동 높이 측정, 스크롤 복원 |

```tsx
import { useVirtualizer } from "@tanstack/react-virtual";

function VirtualList({ items }: { items: Item[] }) {
  const parentRef = useRef<HTMLDivElement>(null);

  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 50, // 항목 높이 추정값 (px)
  });

  return (
    <div
      ref={parentRef}
      style={{ height: 400, overflow: "auto" }}
    >
      <div style={{ height: virtualizer.getTotalSize() }}>
        {virtualizer.getVirtualItems().map((virtualItem) => (
          <div
            key={virtualItem.key}
            style={{
              position: "absolute",
              top: 0,
              left: 0,
              width: "100%",
              height: virtualItem.size,
              transform: `translateY(${virtualItem.start}px)`,
            }}
          >
            {items[virtualItem.index].name}
          </div>
        ))}
      </div>
    </div>
  );
}
```

실전에서는 직접 구현보다 검증된 라이브러리를 쓰는 편이 안전합니다. item 높이가 가변인지, keyboard navigation이 필요한지, sticky header가 있는지에 따라 난도가 올라갑니다.

## state 위치로 렌더링 범위 줄이기

큰 목록과 무관한 state가 목록 부모에 있으면 목록이 자주 다시 계산될 수 있습니다.

```tsx
function Page() {
  const [keyword, setKeyword] = useState("");
  const [isHelpOpen, setIsHelpOpen] = useState(false);

  return (
    <>
      <HelpModal open={isHelpOpen} />
      <SearchInput value={keyword} onChange={setKeyword} />
      <HugeList keyword={keyword} />
    </>
  );
}
```

`isHelpOpen`이 바뀔 때 `HugeList`가 다시 계산될 필요가 없다면 state 위치나 컴포넌트 구조를 조정합니다.

```tsx
function Page() {
  return (
    <>
      <HelpButton /> {/* isHelpOpen state를 내부에 가짐 */}
      <SearchableHugeList /> {/* keyword state를 내부에 가짐 */}
    </>
  );
}
```

최적화는 Hook 하나보다 구조 변경이 더 효과적일 때가 많습니다.

## item 컴포넌트 최적화

목록 항목이 자주 재렌더되고 각 항목이 무겁다면 `memo`로 감쌉니다.

```tsx
const ItemRow = memo(function ItemRow({ item, onSelect }: ItemRowProps) {
  return (
    <div onClick={() => onSelect(item.id)}>
      {item.name}
    </div>
  );
});
```

단, 부모에서 `onSelect`가 매 렌더마다 새 함수라면 `memo`가 무력화됩니다.

```tsx
// onSelect를 useCallback으로 안정화
const handleSelect = useCallback((id: string) => {
  setSelectedId(id);
}, []);
```

또는 item에 id만 넘기고 내부에서 핸들러를 구성합니다.

```tsx
const ItemRow = memo(function ItemRow({
  itemId,
  onSelect,
}: {
  itemId: string;
  onSelect: (id: string) => void;
}) {
  return (
    <div onClick={() => onSelect(itemId)}>
      ...
    </div>
  );
});
```

## deferred value와 큰 목록

검색 input은 즉시 반응하고, 목록 업데이트는 조금 늦게 따라와도 된다면 `useDeferredValue`를 고려합니다.

```tsx
function SearchableLargeList() {
  const [keyword, setKeyword] = useState("");
  const deferredKeyword = useDeferredValue(keyword);
  const isStale = keyword !== deferredKeyword;

  const filtered = useMemo(
    () => largeItems.filter((item) => item.name.includes(deferredKeyword)),
    [deferredKeyword]
  );

  return (
    <>
      <input
        value={keyword}
        onChange={(e) => setKeyword(e.target.value)}
        placeholder="검색"
      />
      <div style={{ opacity: isStale ? 0.6 : 1 }}>
        {filtered.map((item) => (
          <div key={item.id}>{item.name}</div>
        ))}
      </div>
    </>
  );
}
```

이 방식은 API 요청 횟수를 줄이는 debounce가 아닙니다. 느린 렌더링 때문에 입력이 막히는 문제를 완화하는 도구입니다.

## 서버 검색으로 이동하기

목록이 충분히 크거나 필터 조건이 복잡하다면 클라이언트에서 전체 데이터를 들고 있는 것 자체가 문제입니다.

```tsx
function ServerSearch() {
  const [keyword, setKeyword] = useState("");
  const debouncedKeyword = useDebounce(keyword, 300);
  const { data, status } = useFetch<Item[]>(
    debouncedKeyword ? `/api/items?q=${debouncedKeyword}` : null
  );

  return (
    <>
      <input value={keyword} onChange={(e) => setKeyword(e.target.value)} />
      {status === "loading" && <p>검색 중...</p>}
      {status === "success" && (
        <ul>
          {data.map((item) => (
            <li key={item.id}>{item.name}</li>
          ))}
        </ul>
      )}
    </>
  );
}
```

클라이언트에서 10,000개를 필터링하는 것보다 서버에서 20개를 받는 것이 모든 면에서 낫습니다.

## 선택 기준 요약

| 목록 크기 | 특성 | 권장 방향 |
| --- | --- | --- |
| ~100개 | 가벼운 항목 | 그냥 렌더링 |
| ~1,000개 | 계산이 비쌈 | useMemo, 구조 개선 |
| ~1,000개 | 스크롤 중요 | virtualization 검토 |
| 1,000개+ | 페이지 단위 탐색 | pagination |
| 무제한 | 피드, 소셜 | infinite scroll |
| 매우 큰 데이터 | 복잡한 검색 | 서버 검색으로 이동 |

## 읽으면서 생각할 질문

- 느린 원인이 계산, 렌더링, DOM node 수 중 무엇인가?
- 한 화면에 모든 item을 보여줘야 하는가?
- pagination으로 충분한 문제를 virtualization으로 과하게 풀고 있지는 않은가?
- 목록과 무관한 state가 목록 부모에 있어 렌더링 범위를 넓히고 있지는 않은가?
- 사용자가 입력하는 동안 목록이 늦게 따라와도 괜찮은가?
- item마다 무거운 컴포넌트가 있다면 memo와 props 안정화를 함께 했는가?
- 클라이언트 필터링보다 서버 검색이 더 나은 상황은 아닌가?
