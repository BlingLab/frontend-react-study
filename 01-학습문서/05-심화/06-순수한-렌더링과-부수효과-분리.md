# 순수한 렌더링과 부수효과 분리

React 컴포넌트의 렌더링은 순수해야 합니다. 같은 props와 state가 들어오면 같은 JSX를 계산해야 하고, 렌더링 중 외부 세계를 바꾸면 안 됩니다.

이 원칙은 성능과도 연결됩니다. React는 렌더링을 중단하거나 다시 시도하거나 개발 환경에서 한 번 더 호출할 수 있습니다. 렌더링 중 부수효과가 있으면 이런 동작에서 버그가 드러납니다.

## 렌더링 중 하면 안 되는 일

```tsx
function BadComponent({ user }: { user: User }) {
  localStorage.setItem("lastUser", user.id);

  return <p>{user.name}</p>;
}
```

`localStorage` 쓰기는 외부 시스템 변경입니다. 렌더링 중 실행하면 컴포넌트 함수가 다시 호출될 때마다 저장이 발생합니다.

```tsx
function GoodComponent({ user }: { user: User }) {
  useEffect(() => {
    localStorage.setItem("lastUser", user.id);
  }, [user.id]);

  return <p>{user.name}</p>;
}
```

외부 시스템 동기화는 Effect로 보냅니다.

## event handler로 가야 하는 일

사용자 행동의 직접 결과는 event handler에 둡니다.

```tsx
function ExportButton({ report }: { report: Report }) {
  function handleClick() {
    downloadCsv(report);
  }

  return <button onClick={handleClick}>CSV 내보내기</button>;
}
```

버튼을 눌렀기 때문에 다운로드가 일어납니다. 이 로직을 Effect에 넣으면 "어떤 state가 바뀌어서 다운로드되는지"를 추적해야 하고 의도가 흐려집니다.

## Effect로 가야 하는 일

아래는 외부 시스템과 동기화하는 일이므로 Effect가 맞습니다.

- document title 변경
- 브라우저 이벤트 구독
- 타이머 시작과 정리
- 외부 위젯 인스턴스 생성/해제
- localStorage와 state 동기화
- socket 연결

```tsx
useEffect(() => {
  document.title = `${unreadCount}개 알림`;
}, [unreadCount]);
```

## 계산으로 남겨야 하는 일

아래는 Effect도 event handler도 아닙니다. 렌더링 중 계산하면 됩니다.

```tsx
const visibleItems = items.filter((item) => item.title.includes(keyword));
const canSubmit = title.trim().length > 0 && status !== "loading";
const selectedCount = selectedIds.length;
```

계산 가능한 값을 Effect로 state에 다시 저장하면 순수성은 깨지지 않더라도 불필요한 렌더링과 동기화 문제가 생깁니다.

## module scope도 조심하기

컴포넌트 밖 변수에 렌더링 중 값을 쌓으면 여러 컴포넌트 인스턴스가 값을 공유할 수 있습니다.

```tsx
const selectedIds: string[] = [];

function Item({ id }: { id: string }) {
  selectedIds.push(id);

  return <li>{id}</li>;
}
```

렌더링 중 module scope를 바꾸면 순서, 중복, Strict Mode에서 문제가 됩니다. 컴포넌트별 값은 state나 ref, 계산값으로 둡니다.

## 렌더링, 이벤트, Effect 분류 기준

어느 코드가 어디에 가야 할지 헷갈릴 때 이 기준으로 봅니다.

| 종류 | 위치 | 예 |
| --- | --- | --- |
| 계산 | 렌더링 중 | 필터링, 정렬, 요약, 파생 값 |
| 사용자 행동의 직접 결과 | event handler | 저장, 제출, 내보내기, 네비게이션 |
| 외부 시스템 동기화 | Effect | 구독, title 변경, 타이머, socket |

```tsx
function CheckoutPage({ cart }: { cart: CartItem[] }) {
  // 계산 — 렌더링 중
  const total = cart.reduce((sum, item) => sum + item.price * item.quantity, 0);

  // 사용자 행동의 직접 결과 — event handler
  function handlePurchase() {
    submitOrder(cart);
  }

  // 외부 시스템 동기화 — Effect
  useEffect(() => {
    analytics.trackView("checkout");
  }, []);

  return (
    <div>
      <p>합계: {total}원</p>
      <button onClick={handlePurchase}>결제하기</button>
    </div>
  );
}
```

Effect로 가는 코드는 "이 코드가 실행되는 이유가 React 외부 시스템과 동기화하기 때문인가?"로 판단합니다. "버튼을 눌렀기 때문에"가 이유라면 event handler입니다.

## 읽으면서 생각할 질문

- 이 코드는 같은 입력에 대해 같은 JSX만 계산하는가?
- 렌더링 중 외부 시스템을 바꾸고 있지는 않은가?
- 사용자 행동의 결과를 Effect로 옮겨 의도를 흐리고 있지는 않은가?
- 계산 가능한 값을 state로 동기화하고 있지는 않은가?
- module scope를 컴포넌트 state처럼 쓰고 있지는 않은가?
- 이 로직이 렌더링, event handler, Effect 중 어디에 속하는지 판단할 수 있는가?
