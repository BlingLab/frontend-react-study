# 데이터에서 UI로 생각하기

React로 화면을 만들 때 가장 빠른 출발점은 화면 디자인이 아니라 **데이터**입니다. 데이터 모양이 결정되면 컴포넌트 구조와 UI가 자연스럽게 따라옵니다.

## 왜 데이터에서 시작하는가

화면을 먼저 그리고 코드를 맞추려 하면 데이터와 UI 사이의 연결 고리가 흐릿해집니다. 반대로 "이 화면에는 어떤 데이터가 필요한가"를 먼저 생각하면 컴포넌트가 무엇을 받아야 하는지가 자연스럽게 보입니다.

React의 `UI = f(state)` 공식은 이 방향을 설명합니다. 화면은 데이터를 입력으로 받는 함수의 결과이므로, 데이터를 먼저 정하고 그 데이터로 어떤 화면을 만들지 생각하는 흐름이 자연스럽습니다.

## 1단계: 데이터 모양 정하기

학습 대시보드를 만든다고 가정합니다. 먼저 화면이 아니라 데이터를 봅니다.

```ts
type Lesson = {
  id: number;
  title: string;
  level: "기초" | "초급" | "중급";
  completed: boolean;
};

const lessons: Lesson[] = [
  { id: 1, title: "JSX 문법", level: "기초", completed: true },
  { id: 2, title: "컴포넌트 만들기", level: "기초", completed: false },
  { id: 3, title: "State를 고르는 법", level: "초급", completed: false },
];
```

이 데이터만 보고도 화면에 필요한 요소를 추측할 수 있습니다.

- 문서 제목을 보여줘야 합니다.
- 난이도를 보여줄 수 있습니다.
- 완료 여부에 따라 다른 표시가 필요합니다.
- 여러 문서이므로 목록 렌더링이 필요합니다.

데이터 모양을 먼저 적으면 그 다음 단계에서 컴포넌트 props가 자연스럽게 결정됩니다.

## 2단계: 화면을 컴포넌트로 나누기

화면을 보고 어디가 독립적인 컴포넌트가 될 수 있는지 생각합니다. 기준은 단순합니다.

- 같은 구조가 반복되면 컴포넌트로 만들 후보입니다.
- 이름을 붙이면 역할이 더 명확해지는 덩어리가 있다면 후보입니다.
- 데이터 입력이 뚜렷하게 보이면 컴포넌트로 만들기 좋습니다.

학습 대시보드의 경우:

```
StudyDashboard
  ├── LessonSummary    (완료 개수 요약)
  └── LessonList       (목록 전체)
        └── LessonItem (항목 하나)
```

이 구조는 아직 코드가 아닙니다. 화면을 어떻게 나눌지 생각하는 단계입니다. 처음부터 너무 잘게 나누려 하지 않아도 됩니다. 나중에 필요하면 더 나눌 수 있습니다.

## 3단계: 작은 컴포넌트부터 만들기

항목 하나를 먼저 만듭니다. 전체를 한 번에 만들려 하면 복잡해집니다. 작은 것을 완성하고 조합하는 순서가 훨씬 실용적입니다.

```tsx
type LessonItemProps = {
  lesson: Lesson;
};

function LessonItem({ lesson }: LessonItemProps) {
  return (
    <li>
      <strong>{lesson.title}</strong>
      <span> ({lesson.level})</span>
      {lesson.completed ? (
        <span className="badge-done">완료</span>
      ) : (
        <span className="badge-pending">진행 전</span>
      )}
    </li>
  );
}
```

이 컴포넌트는 `lesson` 하나를 받아 `li` 하나를 반환합니다. 입력과 출력이 분명합니다.

## 4단계: 배열은 map으로 목록이 된다

이제 여러 문서를 목록으로 보여줍니다.

```tsx
function LessonList({ lessons }: { lessons: Lesson[] }) {
  return (
    <ul>
      {lessons.map((lesson) => (
        <LessonItem key={lesson.id} lesson={lesson} />
      ))}
    </ul>
  );
}
```

여기서 `map`은 "데이터 하나를 UI 하나로 바꾸는 규칙"입니다. `lessons` 배열에 항목이 10개면 `LessonItem`이 10개 렌더링됩니다. 새 항목이 배열에 추가되면 자동으로 목록에 나타납니다.

`key={lesson.id}`는 React가 각 문서를 안정적으로 구분하게 해줍니다.

## 5단계: 빈 상태도 UI다

목록이 비어 있을 때도 화면은 의도적으로 보여야 합니다. 빈 배열을 렌더링하면 아무것도 안 보이는 것보다, 명확한 메시지를 보여주는 것이 사용자에게 더 좋습니다.

```tsx
function LessonList({ lessons }: { lessons: Lesson[] }) {
  if (lessons.length === 0) {
    return <p className="muted">아직 등록된 학습 문서가 없습니다.</p>;
  }

  return (
    <ul>
      {lessons.map((lesson) => (
        <LessonItem key={lesson.id} lesson={lesson} />
      ))}
    </ul>
  );
}
```

React에서 빈 상태는 예외가 아니라 중요한 UI 상태입니다.

## 6단계: 계산 가능한 값은 직접 계산한다

완료 개수를 보여주고 싶다면 데이터를 기준으로 계산합니다.

```tsx
function LessonSummary({ lessons }: { lessons: Lesson[] }) {
  const completedCount = lessons.filter((lesson) => lesson.completed).length;
  const totalCount = lessons.length;
  const completionRate = totalCount > 0
    ? Math.round((completedCount / totalCount) * 100)
    : 0;

  return (
    <div className="summary">
      <p>총 {totalCount}개 중 {completedCount}개 완료 ({completionRate}%)</p>
    </div>
  );
}
```

`completedCount`, `totalCount`, `completionRate`는 모두 `lessons` 배열에서 계산할 수 있습니다. 이런 값은 별도의 데이터(state)로 들고 있기보다, 필요할 때 렌더링 중 계산하는 편이 단순합니다.

계산 가능한 값을 따로 저장하면 "lessons를 업데이트할 때마다 completedCount도 업데이트해야 한다"는 동기화 문제가 생깁니다.

## 7단계: 전체 화면으로 조립하기

마지막으로 작은 컴포넌트를 조합합니다.

```tsx
function StudyDashboard() {
  return (
    <main>
      <h1>React 학습 현황</h1>
      <LessonSummary lessons={lessons} />
      <LessonList lessons={lessons} />
    </main>
  );
}
```

이제 화면은 아래 흐름으로 읽힙니다.

1. `StudyDashboard`가 전체 구조를 잡습니다.
2. `LessonSummary`가 요약 정보를 보여줍니다.
3. `LessonList`가 배열을 목록으로 바꿉니다.
4. `LessonItem`이 문서 하나를 보여줍니다.

## 데이터 변화에 따른 UI 변화 추론

데이터 중심으로 생각하면 "데이터가 바뀌면 화면이 어떻게 바뀌는가"를 쉽게 추론할 수 있습니다.

```ts
// lessons[1].completed를 true로 바꾸면:
// - LessonItem에서 "진행 전" → "완료"로 바뀜
// - LessonSummary에서 "2개 완료" → "3개 완료"로 바뀜
// - completionRate도 자동으로 다시 계산됨
```

데이터 하나를 바꾸면 그 데이터를 사용하는 모든 화면 조각이 자동으로 업데이트됩니다. 각 컴포넌트를 직접 찾아서 업데이트하는 코드를 작성하지 않아도 됩니다.

## 데이터에서 출발하는 설계 방법 정리

1. 화면에 필요한 데이터 모양을 타입으로 먼저 적습니다.
2. 화면을 컴포넌트 경계로 나눕니다. (작게 시작해도 됩니다.)
3. 가장 작은 컴포넌트(항목 하나)를 먼저 만듭니다.
4. 배열은 `map`으로 목록이 됩니다.
5. 빈 상태, 오류 상태를 항상 같이 생각합니다.
6. 계산 가능한 값은 state로 저장하지 않고 렌더링 중 계산합니다.
7. 작은 컴포넌트를 조합해 전체 화면을 완성합니다.

## 기초 단계에서 다루지 않은 것

이 문서의 예제는 데이터가 고정되어 있습니다. 실제 앱에서는 사용자가 항목을 추가하고, 완료를 토글하고, 삭제합니다. 그러면 데이터가 변하고, 변한 데이터를 React가 알아야 화면을 다시 계산할 수 있습니다.

이 동적인 동작은 `useState`를 배우는 초급 단계에서 다룹니다. 기초 단계에서는 "데이터가 있으면 화면이 어떻게 보이는가"를 먼저 확실히 이해하는 것이 목표입니다.

## 읽으면서 생각할 질문

- 화면에 필요한 데이터 모양을 먼저 말할 수 있는가?
- 데이터 하나를 UI 하나로 바꾸는 컴포넌트가 있는가?
- 배열을 렌더링할 때 안정적인 key를 쓰고 있는가?
- 빈 상태를 의도적으로 보여주고 있는가?
- 계산 가능한 값을 불필요하게 따로 저장하고 있지는 않은가?
- 데이터가 바뀌면 어떤 컴포넌트가 영향을 받는지 추론할 수 있는가?
