# 컴포넌트 만들기

## 컴포넌트란

컴포넌트는 **이름 붙은 UI 단위**입니다. React 앱은 크고 작은 컴포넌트들의 조합으로 이루어집니다.

처음에는 컴포넌트를 "재사용 가능한 것"으로만 생각하기 쉽습니다. 하지만 기초 단계에서는 재사용보다 **읽기 쉬운 덩어리로 나누는 것**이 더 중요합니다. 이름을 붙였을 때 화면의 의미가 더 잘 보이면 좋은 컴포넌트가 될 가능성이 큽니다.

컴포넌트는 대문자로 시작하는 함수입니다. JSX를 반환합니다.

```tsx
function ProfileCard() {
  return (
    <article>
      <h3>Mina</h3>
      <p>Frontend Learner</p>
    </article>
  );
}
```

이 컴포넌트를 다른 곳에서 HTML 태그처럼 쓸 수 있습니다.

```tsx
function App() {
  return (
    <main>
      <ProfileCard />
    </main>
  );
}
```

컴포넌트는 함수이지만, JSX 안에서는 태그처럼 사용합니다.

## 컴포넌트는 순수 함수처럼 동작한다

React 컴포넌트 함수는 **같은 입력(props)에 항상 같은 출력(JSX)을 반환해야 합니다.** 이것을 순수 함수(pure function) 원칙이라고 합니다.

```tsx
// 순수한 컴포넌트 — props만 보고 결과가 결정됨
function Greeting({ name }: { name: string }) {
  return <p>안녕하세요, {name}님!</p>;
}

// Greeting("Mina")는 항상 <p>안녕하세요, Mina님!</p>를 반환
```

이 원칙을 지키면 화면을 예측하기 쉽습니다. "지금 props가 이렇다면 화면은 이렇다"는 관계가 항상 성립합니다.

반대로 컴포넌트가 외부 상태를 바꾸거나 렌더링할 때마다 다른 결과를 낸다면 버그를 찾기 어려워집니다. 이런 부수효과는 `useEffect`나 이벤트 핸들러로 분리하는데, 이는 초급 단계에서 다룹니다.

## 컴포넌트 이름 짓기

좋은 이름은 역할을 바로 드러냅니다. 이름만 보고 어떤 화면 조각인지 알 수 있어야 합니다.

| 좋은 이름 | 피할 이름 | 이유 |
| --- | --- | --- |
| `ProfileCard` | `Card1`, `MyCard` | 숫자나 My는 역할을 숨김 |
| `SkillList` | `List`, `Items` | 너무 일반적 |
| `SearchInput` | `Input2`, `TextBox` | 역할이 불분명 |
| `EmptyMessage` | `Wrapper`, `Box` | 감싸는 역할이라도 이름은 구체적으로 |

컴포넌트 이름은 반드시 **대문자**로 시작합니다. 소문자로 시작하면 React가 HTML 태그로 인식합니다.

```tsx
<profileCard />  // HTML 태그로 인식 → 오류
<ProfileCard />  // 컴포넌트로 인식 → 정상
```

## 컴포넌트 조합하기

작은 컴포넌트를 만들어 더 큰 컴포넌트 안에서 조합합니다.

```tsx
// 작은 단위
function Avatar({ name }: { name: string }) {
  return <div className="avatar">{name[0]}</div>;
}

function SkillTag({ label }: { label: string }) {
  return <span className="tag">{label}</span>;
}

// 조합
function ProfileCard({ name, skills }: { name: string; skills: string[] }) {
  return (
    <article className="card">
      <Avatar name={name} />
      <h3>{name}</h3>
      <div>
        {skills.map((skill) => (
          <SkillTag key={skill} label={skill} />
        ))}
      </div>
    </article>
  );
}
```

이 예제는 화면을 세 가지 책임으로 나눕니다.

- `Avatar`: 이름의 첫 글자를 보여줍니다.
- `SkillTag`: 기술 하나를 태그처럼 보여줍니다.
- `ProfileCard`: 사용자 카드 전체를 조립합니다.

컴포넌트를 나눌 때는 "이 컴포넌트가 어떤 데이터를 받아 어떤 UI를 만드는가"를 같이 생각합니다. 데이터 입력이 분명하면 props도 자연스럽게 정해집니다.

## 한 화면을 컴포넌트로 나누는 과정

아래처럼 한 컴포넌트 안에 모든 UI가 들어 있다고 가정해봅니다.

```tsx
function StudyHome() {
  return (
    <main>
      <section>
        <h2>오늘의 학습</h2>
        <p>기초 단계 진행 중</p>
      </section>

      <section>
        <h2>읽을 문서</h2>
        <ul>
          <li>JSX 문법</li>
          <li>컴포넌트 만들기</li>
          <li>Props 이해하기</li>
        </ul>
      </section>
    </main>
  );
}
```

처음에는 이 정도도 괜찮습니다. 하지만 `오늘의 학습` 영역과 `읽을 문서` 영역이 각각 커지면 이름을 붙여 나눌 수 있습니다.

```tsx
function StudySummary() {
  return (
    <section>
      <h2>오늘의 학습</h2>
      <p>기초 단계 진행 중</p>
    </section>
  );
}

function ReadingList() {
  return (
    <section>
      <h2>읽을 문서</h2>
      <ul>
        <li>JSX 문법</li>
        <li>컴포넌트 만들기</li>
        <li>Props 이해하기</li>
      </ul>
    </section>
  );
}

function StudyHome() {
  return (
    <main>
      <StudySummary />
      <ReadingList />
    </main>
  );
}
```

이제 `StudyHome`은 세부 HTML보다 화면의 큰 구조를 먼저 보여줍니다. 이것이 컴포넌트 분리의 가장 기본적인 장점입니다.

## 언제 컴포넌트를 나눌까

다음 중 하나라도 해당하면 분리를 고려합니다.

**1. 같은 UI 구조가 반복된다**

```tsx
// 반복되는 구조가 보이면 컴포넌트로 추출
function App() {
  return (
    <ul>
      <li>
        <strong>항목 A</strong>
        <span>설명 A</span>
      </li>
      <li>
        <strong>항목 B</strong>
        <span>설명 B</span>
      </li>
    </ul>
  );
}

// ListItem으로 추출
function ListItem({ label, description }: { label: string; description: string }) {
  return (
    <li>
      <strong>{label}</strong>
      <span>{description}</span>
    </li>
  );
}
```

**2. 한 컴포넌트가 너무 많은 역할을 한다**

컴포넌트 함수가 길어지고, 어디서 어디까지가 어떤 UI인지 파악하기 어려워지면 나눌 때가 됐습니다.

**3. 이름을 붙이면 더 읽기 쉬워진다**

분리하는 행위보다 "이름을 붙일 수 있는가"가 더 중요한 기준입니다. `<div className="card-header">` 보다 `<CardHeader>`가 역할을 더 잘 설명한다면 분리할 이유가 있습니다.

**4. 데이터 입력이 뚜렷하다**

아래처럼 이름과 역할만 바뀌는 UI는 props를 받는 컴포넌트로 만들기 좋습니다.

```tsx
function MemberRow({ name, role }: { name: string; role: string }) {
  return (
    <li>
      <strong>{name}</strong>
      <span>{role}</span>
    </li>
  );
}
```

컴포넌트로 나눌 때 "이 컴포넌트의 입력은 무엇인가?"라는 질문이 바로 답이 나오면 좋은 후보입니다.

## 모든 것을 컴포넌트로 만들 필요는 없다

태그 하나짜리 단순한 것까지 전부 컴포넌트로 만들면 오히려 읽기 어려워집니다.

```tsx
// 과도한 분리 — NameText가 없어도 ProfileCard가 충분히 읽힘
function NameText({ name }: { name: string }) {
  return <h3>{name}</h3>;
}

function ProfileCard({ name }: { name: string }) {
  return (
    <article>
      <NameText name={name} />
    </article>
  );
}

// 적당한 수준
function ProfileCard({ name }: { name: string }) {
  return (
    <article>
      <h3>{name}</h3>
    </article>
  );
}
```

## 파일 구조와 export

컴포넌트 하나를 파일 하나에 두는 것이 일반적입니다. 파일 이름은 컴포넌트 이름과 같게 씁니다.

```
ProfileCard.tsx
SkillList.tsx
Button.tsx
```

파일에서 컴포넌트를 내보내는 방법은 두 가지입니다.

```tsx
// 기본 export (default export)
export default function ProfileCard() { ... }

// 이름 있는 export (named export)
export function ProfileCard() { ... }
```

기본 export는 가져올 때 이름을 바꿀 수 있습니다.

```tsx
import ProfileCard from "./ProfileCard";  // default
import AnyName from "./ProfileCard";     // 이름 변경 가능
```

이름 있는 export는 가져올 때 중괄호를 씁니다.

```tsx
import { ProfileCard } from "./ProfileCard";  // named
```

팀마다 선호가 다르지만, 이름 있는 export가 리팩토링 때 더 명확합니다. 파일 이름을 바꾸더라도 컴포넌트 이름이 고정되어 있기 때문입니다.

## 컴포넌트 트리

React 앱은 컴포넌트들이 트리 구조를 이룹니다.

```
App
 ├── Header
 │    └── NavLink (x3)
 ├── MainContent
 │    ├── SearchInput
 │    └── TodoList
 │         └── TodoItem (xN)
 └── Footer
```

부모 컴포넌트가 렌더링되면 그 안의 자식 컴포넌트들도 함께 계산됩니다. 이 관계를 이해하면 나중에 성능 최적화나 상태 위치를 결정할 때 도움이 됩니다.

## 흔한 실수

**컴포넌트 정의를 다른 컴포넌트 안에 넣기**

```tsx
// 나쁜 예: 렌더링할 때마다 Inner가 새로 정의됨
function Outer() {
  function Inner() {   // ← 이렇게 하면 안 됨
    return <p>안쪽</p>;
  }
  return <Inner />;
}

// 올바름: 컴포넌트는 항상 최상위 레벨에서 정의
function Inner() {
  return <p>안쪽</p>;
}

function Outer() {
  return <Inner />;
}
```

`Inner`를 `Outer` 안에 정의하면 `Outer`가 렌더링될 때마다 `Inner`가 새 함수로 만들어집니다. React는 이전 `Inner`와 새 `Inner`를 다른 컴포넌트로 인식하고, 매번 언마운트하고 다시 마운트합니다. state가 있다면 매번 초기화됩니다.

**return 뒤에 줄바꿈만 있는 경우**

```tsx
// 오류: return 뒤에 빈 줄이 있으면 undefined 반환
function Wrong() {
  return
    <div>내용</div>;
}

// 올바름: return 바로 뒤에 (를 쓰거나 JSX를 시작
function Correct() {
  return (
    <div>내용</div>
  );
}
```

## 읽으면서 생각할 질문

- 이 컴포넌트 이름만 보고 화면에서 맡은 역할을 알 수 있는가?
- 이 컴포넌트가 하는 일이 하나인가, 여러 개인가?
- 분리했을 때 코드가 더 읽기 쉬워지는가?
- 소문자로 시작하는 컴포넌트 이름이 있지는 않은가?
- 같은 props가 들어오면 항상 같은 JSX를 반환하는가?
- 컴포넌트 정의를 다른 컴포넌트 안에 넣고 있지는 않은가?
