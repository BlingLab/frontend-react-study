# 접근성 기초

접근성은 스크린리더 사용자만을 위한 것이 아닙니다. 키보드로만 작업하는 사람, 느린 인터넷을 쓰는 사람, 마우스를 쓰지 못하는 상황의 사람 모두를 포함합니다. 접근성이 좋은 UI는 대체로 테스트도 쉽고 유지보수도 좋습니다.

React에서 접근성의 시작은 시맨틱 HTML입니다. 그 위에 ARIA 속성과 focus 관리를 더합니다.

## 시맨틱 HTML 먼저

`div`와 `span`으로만 만든 UI는 보조 기술이 이해하기 어렵습니다. 의미 있는 요소를 먼저 씁니다.

```tsx
// 나쁜 예
<div className="header">
  <div className="nav">
    <div onClick={() => navigate("/home")}>홈</div>
  </div>
</div>

// 좋은 예
<header>
  <nav>
    <a href="/home">홈</a>
  </nav>
</header>
```

| 잘못 쓰기 쉬운 곳 | 맞는 요소 |
| --- | --- |
| 클릭 가능한 div | `<button>` 또는 `<a>` |
| 페이지 상단 | `<header>` |
| 주요 내비게이션 | `<nav>` |
| 주요 콘텐츠 | `<main>` |
| 카드/독립 콘텐츠 | `<article>` |
| 사이드바 | `<aside>` |
| 페이지 하단 | `<footer>` |

`<button>`은 기본으로 키보드 포커스, Enter/Space 클릭, role="button"을 갖습니다. div로 만들면 이것을 모두 직접 구현해야 합니다.

## label과 input 연결

label이 없으면 스크린리더가 input이 무엇인지 알 수 없습니다.

```tsx
// 나쁜 예: placeholder만 있으면 포커스 시 어떤 필드인지 모름
<input type="text" placeholder="이메일" />

// 좋은 예 1: label을 for/htmlFor로 연결
<label htmlFor="email">이메일</label>
<input id="email" type="email" />

// 좋은 예 2: label로 감싸기
<label>
  이메일
  <input type="email" />
</label>

// 좋은 예 3: aria-label (label을 화면에 보여주지 않아야 할 때)
<input type="search" aria-label="제품 검색" />
```

`useId`로 고유 id를 만들면 같은 컴포넌트가 여러 번 렌더링되어도 id 충돌이 없습니다.

```tsx
function SearchField({ label }: { label: string }) {
  const id = useId();

  return (
    <div>
      <label htmlFor={id}>{label}</label>
      <input id={id} type="search" />
    </div>
  );
}
```

## 오류 메시지 연결

오류 메시지는 화면에 보이는 것만으로 충분하지 않습니다. 어떤 input과 연결된 오류인지 보조 기술에 알려야 합니다.

```tsx
function EmailInput() {
  const [value, setValue] = useState("");
  const [error, setError] = useState("");
  const inputId = useId();
  const errorId = useId();

  function handleBlur() {
    if (!value.includes("@")) {
      setError("올바른 이메일 형식을 입력해주세요.");
    } else {
      setError("");
    }
  }

  return (
    <div>
      <label htmlFor={inputId}>이메일</label>
      <input
        id={inputId}
        type="email"
        value={value}
        onChange={(e) => setValue(e.target.value)}
        onBlur={handleBlur}
        aria-describedby={error ? errorId : undefined}
        aria-invalid={error ? true : undefined}
      />
      {error && (
        <p id={errorId} role="alert">
          {error}
        </p>
      )}
    </div>
  );
}
```

- `aria-describedby`: input이 참조하는 설명 요소의 id
- `aria-invalid`: 입력값이 유효하지 않음을 알림
- `role="alert"`: 요소가 나타날 때 스크린리더가 즉시 읽음

## focus 관리

키보드 사용자는 포커스 위치로 맥락을 이해합니다. 화면이 바뀔 때 포커스가 예상하지 못한 곳으로 가거나 사라지면 방향을 잃습니다.

### 모달 열릴 때

모달이 열리면 포커스가 모달 안으로 이동해야 합니다.

```tsx
function Modal({ isOpen, onClose, children }: ModalProps) {
  const closeButtonRef = useRef<HTMLButtonElement>(null);

  useEffect(() => {
    if (isOpen) {
      closeButtonRef.current?.focus();
    }
  }, [isOpen]);

  if (!isOpen) return null;

  return (
    <div role="dialog" aria-modal="true" aria-label="설정">
      <button ref={closeButtonRef} onClick={onClose}>
        닫기
      </button>
      {children}
    </div>
  );
}
```

### 폼 제출 후

제출 성공 후 포커스 위치를 명확히 합니다. 성공 메시지가 있으면 그곳으로, 폼이 초기화되면 첫 번째 input으로 이동합니다.

```tsx
function ContactForm() {
  const [submitted, setSubmitted] = useState(false);
  const successRef = useRef<HTMLParagraphElement>(null);

  async function handleSubmit(event: FormEvent) {
    event.preventDefault();
    await submitForm();
    setSubmitted(true);
  }

  useEffect(() => {
    if (submitted) {
      successRef.current?.focus();
    }
  }, [submitted]);

  if (submitted) {
    return (
      <p ref={successRef} tabIndex={-1} role="status">
        문의가 접수되었습니다.
      </p>
    );
  }

  return <form onSubmit={handleSubmit}>{/* 입력 필드 */}</form>;
}
```

`tabIndex={-1}`은 요소를 tab 순서에서 제외하지만 프로그래밍 방식 focus는 허용합니다.

## 키보드 내비게이션

키보드만으로 모든 기능을 사용할 수 있어야 합니다.

```tsx
function TagList({ tags, onRemove }: { tags: string[]; onRemove: (tag: string) => void }) {
  return (
    <ul>
      {tags.map((tag) => (
        <li key={tag}>
          <span>{tag}</span>
          <button
            type="button"
            onClick={() => onRemove(tag)}
            aria-label={`${tag} 태그 삭제`}
          >
            ×
          </button>
        </li>
      ))}
    </ul>
  );
}
```

`aria-label`이 없으면 "×" 버튼이 여러 개 있을 때 스크린리더 사용자가 어느 태그를 삭제하는지 알 수 없습니다.

드롭다운이나 탭 같은 복잡한 위젯은 WAI-ARIA Authoring Practices를 참고해 Arrow key 내비게이션을 구현합니다.

## 동적 콘텐츠 알림

비동기 결과나 실시간 업데이트는 시각적으로만 바뀌면 스크린리더가 알아차리지 못합니다.

```tsx
function SearchStatus({ status, count }: { status: string; count: number }) {
  return (
    <p role="status" aria-live="polite">
      {status === "loading" && "검색 중..."}
      {status === "success" && `${count}개 결과를 찾았습니다.`}
      {status === "error" && "검색에 실패했습니다."}
    </p>
  );
}
```

- `role="status"` + `aria-live="polite"`: 현재 작업을 방해하지 않고 다음 기회에 읽음
- `role="alert"` + `aria-live="assertive"`: 즉시 읽음 (오류 메시지에 적합)

## 색상과 텍스트

색상만으로 의미를 전달하면 안 됩니다. 색맹 사용자가 구분하지 못할 수 있습니다.

```tsx
// 나쁜 예: 빨간색만으로 오류 표시
<span style={{ color: "red" }}>{error}</span>

// 좋은 예: 아이콘이나 텍스트도 함께
<span style={{ color: "red" }}>
  <span aria-hidden="true">⚠ </span>
  {error}
</span>
```

`aria-hidden="true"`: 보조 기술이 해당 요소를 건너뜁니다. 장식용 아이콘에 씁니다.

## 접근성 테스트

Testing Library의 query 우선순위는 접근성 좋은 UI를 유도합니다.

```tsx
// 좋음: 역할과 이름으로 찾기
screen.getByRole("button", { name: "로그인" });
screen.getByLabelText("이메일");
screen.getByRole("textbox", { name: "검색어" });

// 피할 것: testid로 찾기 (접근성과 무관)
screen.getByTestId("submit-button");
```

`getByRole`이 성공한다면 스크린리더도 같은 방식으로 찾을 수 있다는 뜻입니다.

axe-core나 jest-axe로 자동화된 접근성 검사를 추가할 수 있습니다.

```tsx
import { axe, toHaveNoViolations } from "jest-axe";
expect.extend(toHaveNoViolations);

test("폼에 접근성 오류가 없다", async () => {
  const { container } = render(<ContactForm />);
  const results = await axe(container);
  expect(results).toHaveNoViolations();
});
```

## 접근성 체크리스트

| 항목 | 확인 |
| --- | --- |
| 모든 input에 label이 연결되어 있다 | |
| 오류 메시지가 aria-describedby로 input과 연결되어 있다 | |
| button에 의미 있는 이름이 있다 (aria-label 포함) | |
| 모달 열릴 때 포커스가 모달 안으로 이동한다 | |
| 키보드만으로 모든 기능을 사용할 수 있다 | |
| 색상 외에도 텍스트나 아이콘으로 상태를 표현한다 | |
| 동적 콘텐츠 변화를 aria-live로 알린다 | |
| `getByRole`, `getByLabelText`로 테스트가 가능하다 | |
