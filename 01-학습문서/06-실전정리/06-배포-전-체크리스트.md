# 배포 전 체크리스트

배포 전 체크리스트는 실수를 줄이는 장치입니다. 작은 학습 프로젝트라도 이 습관을 들이면 실전 프로젝트에서 훨씬 안정적으로 마무리할 수 있습니다.

## 기본 명령

```bash
pnpm install
pnpm build
pnpm preview
```

현재 저장소에서는 단계별 미리보기 명령도 확인합니다.

```bash
pnpm run 미리보기:기초
pnpm run 미리보기:중급
pnpm run 미리보기:심화
```

## 기능 체크

| 항목 | 확인 |
| --- | --- |
| 첫 화면 | 빈 화면이나 치명적 오류 없이 열린다. |
| 주요 navigation | 예제 선택이나 URL query가 동작한다. |
| form | 입력, submit, reset, validation이 동작한다. |
| async | loading, error, empty, success를 확인했다. |
| responsive | 모바일 폭에서 텍스트가 겹치지 않는다. |
| accessibility | button 이름, label, error message가 있다. |
| console | 치명적인 runtime error가 없다. |

## 상태별 체크

기능이 정상일 때만 확인하면 실제 사용자가 만나는 실패를 놓칩니다.

| 상태 | 확인 방법 |
| --- | --- |
| loading | 네트워크 지연 또는 mock delay로 직접 봅니다. |
| empty | 빈 배열 응답이나 검색 결과 없음 상태를 만듭니다. |
| error | API 실패, 잘못된 URL, throw로 fallback을 확인합니다. |
| success | 정상 데이터가 화면에 표시됩니다. |
| stale | 이전 데이터 유지 여부가 의도와 맞는지 봅니다. |
| optimistic | 실패 시 rollback 또는 재시도 UI가 보입니다. |

상태별 UI는 코드에 있어도 실제로 눈으로 확인하지 않으면 깨져 있을 가능성이 큽니다.

## 접근성 체크

최소한 아래 항목은 배포 전에 봅니다.

| 항목 | 확인 |
| --- | --- |
| label | input에 label 또는 aria-label이 있습니다. |
| button name | 아이콘 버튼도 접근 가능한 이름이 있습니다. |
| keyboard | Tab, Enter, Escape로 핵심 흐름을 사용할 수 있습니다. |
| focus | modal 열기/닫기 후 focus가 이상한 곳으로 가지 않습니다. |
| error | 오류 메시지는 `role="alert"` 또는 적절한 연결을 갖습니다. |
| aria-busy | 갱신 중인 영역은 필요하면 busy 상태를 알립니다. |

접근성은 "나중에 polish"가 아니라 기능 완료 기준에 포함합니다.

## 문서 체크

README는 배포의 일부입니다.

- 설치 명령이 맞는가?
- 실행 명령이 맞는가?
- 빌드 명령이 맞는가?
- 예제와 연결 문서가 맞는가?
- 새로 추가한 파일이 목차에 연결되어 있는가?
- 외부 자료를 참고했다면 출처가 남아 있는가?

## 환경 변수

실전 프로젝트에서는 환경 변수를 별도로 확인합니다.

```md
## 필요한 환경 변수

- VITE_API_BASE_URL
- VITE_ENABLE_MOCK
```

민감한 값은 README에 실제 값으로 남기지 않습니다. `.env.example`처럼 이름과 형식만 보여줍니다.

## 빌드 결과 확인

개발 서버와 build 결과는 다를 수 있습니다. 배포 전에는 build 결과를 직접 봅니다.

```bash
pnpm build
pnpm preview
```

확인할 것:

- route 또는 query string으로 직접 들어가도 화면이 복원되는가?
- lazy loading chunk가 정상으로 불러와지는가?
- 이미지, CSS, font 경로가 깨지지 않는가?
- 콘솔에 hydration 또는 runtime error가 없는가?
- base path가 있는 배포 환경이면 asset 경로가 맞는가?

Vite 프로젝트는 배포 위치에 따라 `base` 설정이 필요할 수 있습니다. GitHub Pages 같은 하위 경로 배포라면 preview와 실제 배포 경로 차이를 확인합니다.

## 브라우저와 화면 폭

학습 프로젝트라도 최소한 두 폭은 봅니다.

| 환경 | 확인 |
| --- | --- |
| desktop | sidebar, content, table, code block이 안정적으로 보입니다. |
| mobile width | 긴 제목, 버튼, 표가 부모 너비를 넘지 않습니다. |
| keyboard only | 마우스 없이 주요 기능을 사용할 수 있습니다. |
| reduced motion | 애니메이션이 핵심 정보를 가리지 않습니다. |

모바일에서 표가 많은 문서는 가로 스크롤이 생길 수 있습니다. 이때 내용이 잘리는지, 레이아웃 전체가 밀리는지 확인합니다.

## rollback 감각

배포는 올리는 것만이 아니라 되돌릴 수 있어야 합니다.

- 이전 버전을 어떻게 다시 배포할 수 있는가?
- 실패하면 어떤 증상으로 알 수 있는가?
- 빌드 artifact나 배포 로그를 어디에서 확인하는가?
- 배포 후 smoke test는 무엇인가?

학습 프로젝트에서는 간단히 `pnpm build`와 preview 확인으로 시작해도 됩니다.

## PR/커밋 전 최종 확인

커밋하거나 PR을 만들기 전에는 아래를 확인합니다.

```bash
git status --short
git diff --check
pnpm build
```

| 명령 | 목적 |
| --- | --- |
| `git status --short` | 의도하지 않은 파일이 섞였는지 확인 |
| `git diff --check` | trailing whitespace, conflict marker 확인 |
| `pnpm build` | 타입 검사와 production build 확인 |

작업 범위가 문서뿐이어도 링크와 예제 앱이 깨지지 않았는지 build로 확인합니다.

## 배포 기록 템플릿

배포나 PR 설명에 아래 형식으로 남깁니다.

```md
## 변경 내용

- 변경한 기능이나 문서

## 확인한 것

- [ ] pnpm build
- [ ] 주요 화면 수동 확인
- [ ] loading/error/empty 상태 확인
- [ ] 모바일 폭 확인

## 남은 위험

- 남아 있는 제한 사항

## rollback

- 이전 커밋/태그로 되돌릴 수 있음
```

## 읽으면서 생각할 질문

- README만 보고 새 사람이 실행할 수 있는가?
- 새 문서나 예제가 목차에서 빠지지 않았는가?
- 실패 상태를 실제로 눈으로 확인했는가?
- 모바일 폭에서 UI가 깨지지 않는가?
- 배포 실패 시 되돌릴 방법을 알고 있는가?
- build 결과와 개발 서버 결과를 모두 확인했는가?
- 커밋에 의도하지 않은 파일이 섞이지 않았는가?
