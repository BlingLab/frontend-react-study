# 면접 답변으로 정리하기

면접에서는 React API 정의를 외우는 것보다, 내가 왜 그런 구조를 선택했는지 설명하는 힘이 중요합니다.

좋은 답변은 보통 이런 순서를 가집니다.

1. 문제 상황
2. 고려한 선택지
3. 선택한 방식
4. 코드에서 얻은 결과
5. 다음에 개선할 점

예를 들어 Context를 썼다면 "props drilling이 싫어서"에서 끝내지 않습니다. 어떤 값이 여러 컴포넌트에서 필요했고, Provider 범위를 어디로 잡았으며, 자주 바뀌는 값 때문에 어떤 점을 조심했는지 말합니다.

## 주제별 답변 뼈대

### state 위치

```md
이 값은 여러 컴포넌트가 함께 봐야 했기 때문에 가장 가까운 공통 부모로 올렸습니다.
자식은 값을 직접 바꾸지 않고 `onToggle` 같은 handler를 호출하게 했습니다.
이렇게 하니 source of truth가 하나가 되어 완료 개수와 목록 표시가 어긋나지 않았습니다.
```

### Effect

```md
이 Effect는 렌더링 계산이 아니라 외부 시스템과 동기화하기 위한 코드였습니다.
API 요청/이벤트 구독/타이머처럼 React 바깥의 일이었고, cleanup이 필요한 경우에는 return에서 정리했습니다.
계산 가능한 값은 Effect로 state에 저장하지 않고 렌더링 중 계산했습니다.
```

### 성능

```md
처음부터 memoization을 넣지 않고 Profiler로 느린 행동을 확인했습니다.
병목이 큰 목록 필터링에 있었기 때문에 계산 결과를 `useMemo`로 캐시했고,
목록 크기가 더 커질 경우 pagination이나 virtualization을 다음 개선으로 생각했습니다.
```

### 테스트

```md
구현 함수보다 사용자가 보는 결과를 기준으로 테스트했습니다.
사용자가 입력하고 버튼을 누르면 목록에 항목이 추가되는지,
실패하면 오류 메시지와 다시 시도 버튼이 보이는지를 검증했습니다.
```

## 자주 나오는 질문과 답변 방향

### "React에서 state와 props의 차이가 무엇인가요?"

- props는 부모에서 전달되는 읽기 전용 값, state는 컴포넌트 안에서 바뀌는 값입니다.
- 덧붙이면: "동일한 값이라도 누가 소유하느냐에 따라 state가 될 수도 있고 props가 될 수도 있습니다. source of truth를 한 곳에 두고 내려주는 것이 핵심입니다."

### "useEffect를 언제 써야 하나요?"

기초 답변: "외부 시스템과 동기화할 때 씁니다."

심화 답변:
```md
Effect는 escape hatch입니다. React 바깥의 시스템, 즉 브라우저 API, 네트워크,
타이머, 외부 라이브러리와 동기화할 때 씁니다.
props나 state에서 계산 가능한 값은 Effect가 아니라 렌더링 중에 계산합니다.
클릭 핸들러로 처리할 수 있는 로직도 Effect가 아닙니다.
저는 먼저 Effect를 제거할 방법을 찾고, 그게 안 될 때 Effect를 씁니다.
```

### "Context와 상태 관리 라이브러리의 차이는 무엇인가요?"

```md
Context는 값을 트리 아래로 전달하는 통로입니다. 상태 관리 도구가 아닙니다.
자주 바뀌지 않는 값, 예를 들어 테마나 로그인 사용자 정보를 여러 컴포넌트에
전달할 때 잘 맞습니다.
자주 바뀌는 값에 Context를 쓰면 Provider 아래 모든 컴포넌트가 다시 렌더링됩니다.
이 경우 Zustand, Jotai 같은 도구가 더 효율적입니다.
저는 상태 관리 라이브러리를 먼저 선택하기보다, 상태의 성격을 먼저 판단합니다.
```

### "key는 왜 써야 하나요?"

```md
React가 목록을 효율적으로 재조정하기 위해 씁니다.
key가 없으면 인덱스로 위치를 판단하는데, 순서가 바뀌면 state가 섞입니다.
key는 목록 식별 외에 특정 화면의 state를 초기화하는 데도 씁니다.
예를 들어 선택한 사용자 ID를 key로 주면 사용자가 바뀔 때 상세 컴포넌트가
새로 마운트되어 이전 state가 남지 않습니다.
```

### "useMemo와 useCallback은 항상 써야 하나요?"

```md
아니오. 모든 계산과 함수에 memoization을 붙이면 코드가 복잡해지고
오히려 성능이 나빠질 수 있습니다.
저는 먼저 측정합니다. 큰 목록 필터링처럼 실제로 느린 계산이 있거나,
자식 컴포넌트가 memo로 감싸져 있는데 참조가 매 렌더마다 바뀔 때 씁니다.
필요성이 증명되기 전에 먼저 useMemo를 붙이는 것은 피합니다.
```

### "Error Boundary는 어떻게 동작하나요?"

```md
렌더링 중 자식 컴포넌트가 throw하면 부모 Error Boundary가 이를 잡고
fallback UI를 보여줍니다.
event handler나 비동기 콜백에서 발생한 오류는 잡지 못합니다. 그런 오류는
try/catch로 직접 처리하고 state로 보여줍니다.
저는 Error Boundary를 앱 최상단뿐 아니라 독립적인 위젯 단위로 두어
한 영역이 실패해도 다른 영역이 계속 동작하도록 합니다.
```

### "React 19에서 달라진 점은 무엇인가요?"

```md
useOptimistic과 useActionState가 안정 API로 들어왔습니다.
useOptimistic은 서버 응답을 기다리지 않고 UI를 먼저 바꾼 뒤
실패하면 이전 상태로 되돌립니다.
use(Promise)로 Promise를 렌더링 중에 읽을 수 있게 되었고,
이 경우 Suspense가 loading 경계가 됩니다.
ref를 JSX에서 직접 prop으로 전달할 수 있게 되어 forwardRef가 필요 없어졌습니다.
```

## 답변을 망치는 패턴

- API 정의만 말하고 실제 코드 경험이 없다.
- "성능 때문에 useMemo를 썼다"면서 측정 근거가 없다.
- Context를 "전역 상태 관리 도구"라고만 설명한다.
- Effect를 "마운트 때 실행하려고" 썼다고만 말한다.
- 실패 상태나 빈 상태를 고려하지 않았다.
- "공식 문서대로 했다"에서 멈추고 내가 겪은 판단 과정을 말하지 않는다.

면접 답변은 개념 암기가 아니라 판단 과정의 압축입니다.

## 준비 방법

각 구조 선택에 대해 다음 세 가지를 말할 수 있으면 됩니다.

| 질문 | 예시 |
| --- | --- |
| 언제 선택하는가? | "자주 바뀌지 않는 값을 여러 컴포넌트에 전달할 때" |
| 다른 선택지는 무엇인가? | "props drilling, 상태 관리 라이브러리" |
| 선택하지 않는 경우는? | "자주 바뀌는 값이면 Context보다 다른 도구를 먼저 봅니다" |

## 읽으면서 생각할 질문

- 이 코드에서 가장 중요한 상태는 무엇인가?
- state 위치를 왜 그렇게 정했는가?
- Effect를 쓴 곳은 어떤 외부 시스템과 연결되는가?
- reducer나 context를 썼다면 복잡도가 실제로 줄었는가?
- 다음에 다시 만들면 어떤 구조를 바꿀 것인가?
- 가장 최근에 내린 판단 중 설명할 수 있는 것은 무엇인가?
