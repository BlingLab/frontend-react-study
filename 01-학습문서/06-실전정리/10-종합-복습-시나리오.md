# 종합 복습 시나리오

기초부터 심화까지 배운 개념들이 실제 기능에서 어떻게 연결되는지 확인합니다. 한 기능을 처음부터 끝까지 설계하고 구현하는 흐름을 시나리오로 따라가며, 각 단계에서 어떤 판단을 했는지 말로 설명할 수 있는지 점검합니다.

## 시나리오: 게시글 검색 기능 만들기

사용자가 키워드를 입력하면 API로 게시글을 검색하고, 결과를 목록으로 보여주는 기능입니다.

완료 조건을 먼저 적습니다.

```md
## 완료 조건

- 사용자가 검색어를 입력하면 검색 버튼을 누를 수 있다.
- 검색 중에는 "검색 중..." 문구가 보이고 버튼이 비활성화된다.
- 결과가 있으면 목록이 보인다.
- 결과가 없으면 "결과가 없습니다." 문구가 보인다.
- 실패하면 오류 메시지와 다시 시도 버튼이 보인다.
- 검색어를 비우면 초기 상태로 돌아간다.
- 키보드만으로 검색을 완료할 수 있다.
- pnpm build가 통과한다.
```

## 1단계: 타입과 상태 설계

화면부터 만들기 전에 데이터 타입과 상태를 설계합니다.

```tsx
type Post = {
  id: string;
  title: string;
  author: string;
  createdAt: string;
};

type SearchPhase =
  | { phase: "idle" }
  | { phase: "loading"; keyword: string }
  | { phase: "success"; keyword: string; results: Post[] }
  | { phase: "error"; keyword: string; message: string };
```

설명할 수 있어야 할 것:

- `SearchPhase`는 discriminated union입니다. `phase`로 좁히면 각 단계에서만 있는 필드에 접근할 수 있습니다.
- `loading` 상태에 `results`가 없고, `success`에 `message`가 없으므로 잘못된 상태 조합이 타입으로 막힙니다.
- `Post` 타입은 API 응답 형태와 일치해야 합니다.

## 2단계: 최소 UI 만들기

상태 연결 전에 UI 껍데기를 먼저 만듭니다.

```tsx
function PostSearch() {
  return (
    <div>
      <form>
        <label htmlFor="search-input">검색어</label>
        <input id="search-input" type="search" placeholder="게시글 제목 검색" />
        <button type="submit">검색</button>
      </form>
      <p>검색어를 입력하세요.</p>
    </div>
  );
}
```

설명할 수 있어야 할 것:

- `label`과 `htmlFor`로 input과 label을 연결했습니다.
- `type="search"`는 시맨틱 HTML입니다.
- `type="submit"` 버튼은 Enter 키로도 submit됩니다.

## 3단계: state와 event handler 연결

```tsx
function PostSearch() {
  const [keyword, setKeyword] = useState("");
  const [searchPhase, setSearchPhase] = useState<SearchPhase>({ phase: "idle" });

  async function handleSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault();

    if (!keyword.trim()) return;

    setSearchPhase({ phase: "loading", keyword });

    try {
      const results = await fetchPosts(keyword);
      setSearchPhase({ phase: "success", keyword, results });
    } catch (err) {
      const message = err instanceof Error ? err.message : "알 수 없는 오류";
      setSearchPhase({ phase: "error", keyword, message });
    }
  }

  return (
    <div>
      <form onSubmit={handleSubmit}>
        <label htmlFor="search-input">검색어</label>
        <input
          id="search-input"
          type="search"
          value={keyword}
          onChange={(e) => setKeyword(e.target.value)}
          placeholder="게시글 제목 검색"
        />
        <button type="submit" disabled={searchPhase.phase === "loading"}>
          {searchPhase.phase === "loading" ? "검색 중..." : "검색"}
        </button>
      </form>
      <SearchResult phase={searchPhase} onRetry={() => handleSubmit(/* ... */)} />
    </div>
  );
}
```

설명할 수 있어야 할 것:

- `keyword`는 controlled input입니다. 검색어가 다른 곳에서도 필요하면 위로 올릴 수 있습니다.
- `searchPhase`는 loading, success, error, idle 네 상태를 하나로 관리합니다.
- `disabled={searchPhase.phase === "loading"}`으로 중복 제출을 막습니다.
- `event.preventDefault()`가 없으면 form submit 시 페이지가 새로고침됩니다.

## 4단계: 결과 컴포넌트 분리

```tsx
type SearchResultProps = {
  phase: SearchPhase;
  onRetry: () => void;
};

function SearchResult({ phase, onRetry }: SearchResultProps) {
  if (phase.phase === "idle") {
    return <p>검색어를 입력하세요.</p>;
  }

  if (phase.phase === "loading") {
    return <p aria-live="polite">검색 중...</p>;
  }

  if (phase.phase === "error") {
    return (
      <div role="alert">
        <p>검색에 실패했습니다: {phase.message}</p>
        <button type="button" onClick={onRetry}>
          다시 시도
        </button>
      </div>
    );
  }

  if (phase.results.length === 0) {
    return <p>"{phase.keyword}"에 대한 결과가 없습니다.</p>;
  }

  return (
    <ul>
      {phase.results.map((post) => (
        <PostItem key={post.id} post={post} />
      ))}
    </ul>
  );
}

function PostItem({ post }: { post: Post }) {
  return (
    <li>
      <h3>{post.title}</h3>
      <p>
        {post.author} · {post.createdAt}
      </p>
    </li>
  );
}
```

설명할 수 있어야 할 것:

- `SearchResult`는 `phase`를 props로 받아 상태별 UI만 담당합니다.
- discriminated union 덕분에 `phase.phase === "error"` 이후 `phase.message`에 안전하게 접근합니다.
- `role="alert"`는 오류가 나타날 때 스크린리더가 바로 읽습니다.
- `PostItem`을 분리해 목록 항목이 복잡해져도 `SearchResult`가 복잡해지지 않습니다.

## 5단계: Custom Hook으로 분리

상태 로직과 UI를 분리합니다.

```tsx
function usePostSearch() {
  const [keyword, setKeyword] = useState("");
  const [phase, setPhase] = useState<SearchPhase>({ phase: "idle" });

  async function search(inputKeyword: string) {
    if (!inputKeyword.trim()) return;

    setPhase({ phase: "loading", keyword: inputKeyword });

    try {
      const results = await fetchPosts(inputKeyword);
      setPhase({ phase: "success", keyword: inputKeyword, results });
    } catch (err) {
      const message = err instanceof Error ? err.message : "알 수 없는 오류";
      setPhase({ phase: "error", keyword: inputKeyword, message });
    }
  }

  function retry() {
    if (phase.phase === "error") {
      search(phase.keyword);
    }
  }

  return { keyword, setKeyword, phase, search, retry };
}

function PostSearch() {
  const { keyword, setKeyword, phase, search, retry } = usePostSearch();

  function handleSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault();
    search(keyword);
  }

  return (
    <div>
      <form onSubmit={handleSubmit}>
        <label htmlFor="search-input">검색어</label>
        <input
          id="search-input"
          type="search"
          value={keyword}
          onChange={(e) => setKeyword(e.target.value)}
          placeholder="게시글 제목 검색"
        />
        <button type="submit" disabled={phase.phase === "loading"}>
          {phase.phase === "loading" ? "검색 중..." : "검색"}
        </button>
      </form>
      <SearchResult phase={phase} onRetry={retry} />
    </div>
  );
}
```

설명할 수 있어야 할 것:

- `usePostSearch`는 상태 관리와 API 호출 로직을 담습니다.
- 컴포넌트는 "어떻게 보여줄지"만 담당합니다.
- `retry`는 hook 안에서 마지막으로 검색한 keyword를 알고 있으므로 컴포넌트에서는 retry 기능을 단순하게 쓸 수 있습니다.

## 6단계: 테스트 시나리오 작성

```tsx
describe("PostSearch", () => {
  const mockFetch = vi.fn();
  global.fetch = mockFetch;

  describe("검색 성공", () => {
    test("결과가 있으면 목록이 보인다", async () => {
      mockFetch.mockResolvedValueOnce({
        ok: true,
        json: async () => [{ id: "1", title: "React Hooks", author: "mina", createdAt: "2025-01-01" }],
      });

      const user = userEvent.setup();
      render(<PostSearch />);

      await user.type(screen.getByLabelText("검색어"), "react");
      await user.click(screen.getByRole("button", { name: "검색" }));

      expect(await screen.findByText("React Hooks")).toBeInTheDocument();
    });

    test("결과가 없으면 빈 상태 문구가 보인다", async () => {
      mockFetch.mockResolvedValueOnce({ ok: true, json: async () => [] });

      const user = userEvent.setup();
      render(<PostSearch />);

      await user.type(screen.getByLabelText("검색어"), "없는검색어");
      await user.click(screen.getByRole("button", { name: "검색" }));

      expect(await screen.findByText(/결과가 없습니다/)).toBeInTheDocument();
    });
  });

  describe("검색 중", () => {
    test("버튼이 비활성화되고 loading 문구가 보인다", async () => {
      let resolve: (v: unknown) => void;
      mockFetch.mockReturnValueOnce(new Promise((res) => { resolve = res; }));

      const user = userEvent.setup();
      render(<PostSearch />);

      await user.type(screen.getByLabelText("검색어"), "react");
      await user.click(screen.getByRole("button", { name: "검색" }));

      expect(screen.getByRole("button", { name: "검색 중..." })).toBeDisabled();

      resolve!({ ok: true, json: async () => [] });
    });
  });

  describe("검색 실패", () => {
    test("오류 메시지와 다시 시도 버튼이 보인다", async () => {
      mockFetch.mockRejectedValueOnce(new Error("네트워크 오류"));

      const user = userEvent.setup();
      render(<PostSearch />);

      await user.type(screen.getByLabelText("검색어"), "react");
      await user.click(screen.getByRole("button", { name: "검색" }));

      expect(await screen.findByRole("alert")).toBeInTheDocument();
      expect(screen.getByRole("button", { name: "다시 시도" })).toBeInTheDocument();
    });
  });
});
```

설명할 수 있어야 할 것:

- `getByLabelText("검색어")`는 label과 input이 올바르게 연결됐을 때만 동작합니다.
- `getByRole("button", { name: "검색" })`은 버튼 이름으로 찾습니다.
- 검색 중 테스트에서 resolve를 뒤에 호출해 loading 상태를 관찰합니다.
- `findByText`는 async로 요소가 나타날 때까지 기다립니다.

## 면접 답변 연습

이 기능을 만든 후 아래 질문에 답해봅니다.

**Q. 검색 상태를 왜 discriminated union으로 만들었나요?**

`status: string`으로 하면 `loading` 상태에서 `results`에 접근하는 실수를 컴파일 단계에서 막지 못합니다. discriminated union을 쓰면 `phase === "success"` 이후에만 `results`에 접근할 수 있어서, 타입이 상태 조합의 불변식을 보장합니다.

**Q. Custom Hook을 왜 만들었나요?**

컴포넌트가 API 호출 로직, 상태 전환, UI 렌더링을 모두 하면 테스트하기 어렵고 로직 재사용도 불가능합니다. `usePostSearch`로 분리하면 컴포넌트는 화면만 담당하고, hook은 `renderHook`으로 UI 없이 테스트할 수 있습니다.

**Q. 접근성을 위해 무엇을 했나요?**

`htmlFor`로 label과 input을 연결했고, 오류 메시지에 `role="alert"`를 붙여 스크린리더가 바로 읽도록 했습니다. 검색 버튼에는 상태에 따라 이름이 바뀌도록 했고(`검색` → `검색 중...`), `disabled`로 로딩 중 중복 제출을 막았습니다.
