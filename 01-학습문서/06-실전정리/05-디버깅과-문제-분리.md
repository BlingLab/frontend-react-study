# 디버깅과 문제 분리

React에서 버그를 만났을 때 "React가 이상하다"로 시작하면 오래 걸립니다. 문제를 종류별로 나누면 원인을 훨씬 빨리 좁힐 수 있습니다.

## 문제 분류표

| 증상 | 먼저 의심할 것 |
| --- | --- |
| 화면이 안 바뀜 | state를 직접 수정했는가, setter를 호출했는가 |
| 값이 한 박자 늦음 | state snapshot을 이해하지 못했는가 |
| 입력값이 초기화됨 | key가 바뀌거나 컴포넌트 위치가 바뀌었는가 |
| Effect가 계속 실행됨 | dependency에 매번 새 객체/함수가 들어가는가 |
| 목록 item state가 섞임 | key가 index이거나 불안정한가 |
| API 결과가 뒤바뀜 | 오래된 응답이 최신 요청을 덮어쓰는가 |
| 버튼이 여러 번 실행됨 | loading 중 disabled 처리가 없는가 |
| 스타일이 겹침 | layout 제약, width, overflow, z-index 문제인가 |

## 재현을 먼저 고정하기

버그를 고치기 전 재현 절차를 적습니다.

```md
1. 중급 예제를 연다.
2. 검색어에 react를 입력한다.
3. 카테고리를 Hook으로 바꾼다.
4. 초기화를 누른다.
5. input focus가 돌아오지 않는다.
```

재현 절차가 없으면 고친 것처럼 보여도 다시 깨질 수 있습니다.

## console.log를 구조적으로 쓰기

무작정 로그를 많이 찍기보다 흐름을 따라 찍습니다.

```tsx
console.log("render", { keyword, selectedCategory });
console.log("submit", { keyword });
console.log("effect start", { url });
console.log("effect cleanup", { url });
```

렌더링, event, Effect를 구분해서 보면 문제가 어디에서 시작되는지 보입니다.

## React DevTools로 볼 것

- props가 실제로 내려오는가?
- state가 기대한 값인가?
- 같은 컴포넌트가 unmount/remount되고 있지는 않은가?
- context value가 너무 자주 바뀌는가?
- Profiler에서 어떤 컴포넌트가 느린가?

DevTools는 console보다 컴포넌트 경계를 보기 좋습니다.

## 최소 재현 만들기

복잡한 앱에서 바로 고치기 어렵다면 작은 컴포넌트로 분리합니다.

```tsx
function Repro() {
  const [items, setItems] = useState(initialItems);

  return <ProblemList items={items} onChange={setItems} />;
}
```

문제가 재현되는 최소 코드가 만들어지면 원인도 대체로 좁혀집니다.

## 읽으면서 생각할 질문

- 이 버그를 5단계 이하로 재현할 수 있는가?
- 문제는 render, event handler, Effect, API, CSS 중 어디에 가까운가?
- state를 직접 변경하고 있지는 않은가?
- key나 컴포넌트 위치 때문에 state가 초기화되는가?
- 작은 재현 컴포넌트로 문제를 분리할 수 있는가?
