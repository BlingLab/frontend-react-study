# 디버깅과 문제 분리

React에서 버그를 만났을 때 "React가 이상하다"로 시작하면 오래 걸립니다. 문제를 종류별로 나누면 원인을 훨씬 빨리 좁힐 수 있습니다.

## 문제 분류표

| 증상 | 먼저 의심할 것 |
| --- | --- |
| 화면이 안 바뀜 | state를 직접 수정했는가, setter를 호출했는가 |
| 값이 한 박자 늦음 | state snapshot을 이해하지 못했는가 |
| 입력값이 초기화됨 | key가 바뀌거나 컴포넌트 위치가 바뀌었는가 |
| Effect가 계속 실행됨 | dependency에 매번 새 객체/함수가 들어가는가 |
| 목록 item state가 섞임 | key가 index이거나 불안정한가 |
| API 결과가 뒤바뀜 | 오래된 응답이 최신 요청을 덮어쓰는가 (race condition) |
| 버튼이 여러 번 실행됨 | loading 중 disabled 처리가 없는가 |
| 스타일이 겹침 | layout 제약, width, overflow, z-index 문제인가 |
| 함수가 오래된 값을 읽음 | Effect 또는 callback 안에서 클로저가 stale한가 |
| Context 변경이 퍼지지 않음 | Provider가 올바른 트리에 있는가 |

## 재현을 먼저 고정하기

버그를 고치기 전 재현 절차를 적습니다.

```md
1. 중급 예제를 연다.
2. 검색어에 react를 입력한다.
3. 카테고리를 Hook으로 바꾼다.
4. 초기화를 누른다.
5. input focus가 돌아오지 않는다.
```

재현 절차가 없으면 고친 것처럼 보여도 다시 깨질 수 있습니다.

## console.log를 구조적으로 쓰기

무작정 로그를 많이 찍기보다 흐름을 따라 찍습니다.

```tsx
console.log("render", { keyword, selectedCategory });
console.log("submit", { keyword });
console.log("effect start", { url });
console.log("effect cleanup", { url });
```

렌더링, event, Effect를 구분해서 보면 문제가 어디에서 시작되는지 보입니다.

렌더링 중 console.log를 찍으면 Strict Mode에서 두 번 찍힐 수 있습니다. 이는 정상이며 개발 환경에서만 발생합니다.

## React DevTools로 볼 것

- props가 실제로 내려오는가?
- state가 기대한 값인가?
- 같은 컴포넌트가 unmount/remount되고 있지는 않은가?
- context value가 너무 자주 바뀌는가?
- Profiler에서 어떤 컴포넌트가 느린가?
- "왜 렌더링됐나" (Highlight updates 기능)

DevTools는 console보다 컴포넌트 경계를 보기 좋습니다.

## stale closure 패턴 인식하기

Effect나 timeout 안에서 오래된 state 값을 읽는 패턴입니다.

```tsx
// 버그: handleClick이 생성될 때의 count값을 기억함
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log(count); // 항상 0을 출력 — stale closure
    }, 1000);
    return () => clearInterval(id);
  }, []); // count가 dependency에 없음

  return <button onClick={() => setCount((c) => c + 1)}>{count}</button>;
}
```

해결 방법 1: dependency에 추가합니다.

```tsx
useEffect(() => {
  const id = setInterval(() => {
    console.log(count);
  }, 1000);
  return () => clearInterval(id);
}, [count]); // count가 바뀔 때 다시 구독
```

해결 방법 2: setter의 함수 형태를 씁니다.

```tsx
useEffect(() => {
  const id = setInterval(() => {
    setCount((c) => c + 1); // 현재 state를 함수 인수로 받음
  }, 1000);
  return () => clearInterval(id);
}, []); // count에 의존하지 않아도 됨
```

## 최소 재현 만들기

복잡한 앱에서 바로 고치기 어렵다면 작은 컴포넌트로 분리합니다.

```tsx
function Repro() {
  const [items, setItems] = useState(initialItems);

  return <ProblemList items={items} onChange={setItems} />;
}
```

문제가 재현되는 최소 코드가 만들어지면 원인도 대체로 좁혀집니다.

최소 재현이 유용한 경우:
- 팀에 질문할 때
- Stack Overflow나 GitHub Issue에 올릴 때
- 문제 범위를 좁히는 과정 자체에서 원인을 발견할 때

## TypeScript가 알려주는 버그 신호

타입 오류가 런타임 버그로 이어지는 경우가 많습니다.

```tsx
// 타입 오류: results가 undefined일 수 있는데 .map()을 바로 호출
function ResultList({ data }: { data: Post[] | undefined }) {
  return data.map(...);  // Cannot read properties of undefined
}

// 수정
function ResultList({ data }: { data: Post[] | undefined }) {
  if (!data) return null;
  return data.map(...);
}
```

`as any`나 `!` 단언이 많은 코드는 런타임 오류가 숨어있을 가능성이 높습니다. 타입 오류를 무시하는 코드가 있다면 그 주변부터 의심합니다.

## Effect dependency 확인하기

Effect가 의도보다 많이 실행되거나 실행되지 않는 문제는 dependency 배열부터 봅니다.

```tsx
useEffect(() => {
  fetchData(url);
}, []);  // url이 빠져 있으면 url이 바뀌어도 다시 fetch하지 않음
```

ESLint `react-hooks/exhaustive-deps` 규칙이 빠진 dependency를 알려줍니다. 이 경고를 무시하면 Effect가 오래된 값을 참조하는 클로저 버그가 생깁니다.

**Effect가 너무 자주 실행될 때:**

```tsx
useEffect(() => {
  fetch(`/api/items?page=${page}`);
}, [page, { sort: "asc" }]); // 객체 리터럴 — 매 렌더마다 새 참조

// 수정: 원시값만 dependency에 두거나 useMemo로 안정화
useEffect(() => {
  fetch(`/api/items?page=${page}&sort=asc`);
}, [page]);
```

## API 응답 경쟁 조건 (race condition)

빠른 입력에서 느린 요청과 빠른 요청의 응답 순서가 뒤바뀔 수 있습니다.

```tsx
// 버그: 마지막 요청의 응답이 먼저 올 수 있음
useEffect(() => {
  fetch(`/api/search?q=${keyword}`)
    .then((res) => res.json())
    .then((data) => setResults(data));
}, [keyword]);

// 수정: ignore flag로 오래된 요청 응답 무시
useEffect(() => {
  let ignore = false;

  fetch(`/api/search?q=${keyword}`)
    .then((res) => res.json())
    .then((data) => {
      if (!ignore) setResults(data);
    });

  return () => {
    ignore = true; // 새 요청이 시작되면 이전 응답 무시
  };
}, [keyword]);
```

## 디버깅 단계 흐름

1. 증상을 정확히 묘사합니다. ("버튼을 누르면 X가 되어야 하는데 Y가 된다")
2. 재현 절차를 5단계 이하로 좁힙니다.
3. 문제 종류를 render/event/Effect/API/CSS로 분류합니다.
4. 분류에 맞는 도구를 씁니다.
   - render: DevTools Component 탭, console.log
   - event: 이벤트 핸들러에 console.log, DevTools Events
   - Effect: dependency 확인, cleanup 확인
   - API: Network 탭, ignore flag 확인
   - CSS: Elements 탭, computed 스타일
5. 최소 재현 코드로 줄입니다.
6. 고친 뒤 재현 절차로 다시 확인합니다.

## 읽으면서 생각할 질문

- 이 버그를 5단계 이하로 재현할 수 있는가?
- 문제는 render, event handler, Effect, API, CSS 중 어디에 가까운가?
- state를 직접 변경하고 있지는 않은가?
- key나 컴포넌트 위치 때문에 state가 초기화되는가?
- 작은 재현 컴포넌트로 문제를 분리할 수 있는가?
- Effect dependency 배열에서 빠진 값이 없는가?
- TypeScript 타입 오류나 any 단언이 버그의 근원인가?
- 비동기 응답이 경쟁 조건으로 뒤바뀔 수 있는가?
- stale closure가 오래된 값을 참조하고 있지 않은가?
