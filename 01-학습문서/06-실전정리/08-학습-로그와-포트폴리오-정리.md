# 학습 로그와 포트폴리오 정리

React를 공부했다는 사실보다 중요한 것은 무엇을 만들었고, 어떤 판단을 했고, 무엇을 다시 설명할 수 있는가입니다. 학습 로그는 이 과정을 남기는 문서입니다.

## 학습 로그 템플릿

```md
## 날짜: YYYY-MM-DD

- 학습 주제:
- 읽은 문서:
- 직접 써본 코드:
- 새로 이해한 개념:
- 아직 헷갈리는 점:
- 다음에 확인할 것:
```

짧아도 꾸준히 남기면 면접 답변과 포트폴리오 설명의 재료가 됩니다.

### 학습 로그 예시

```md
## 2024-03-15

- 학습 주제: useEffect dependency와 cleanup
- 읽은 문서: 03-중급/01-effect를-생각하는법.md
- 직접 써본 코드: EffectsFetchingExample에서 ignore flag 패턴 추가
- 새로 이해한 개념: ignore flag가 없으면 컴포넌트 unmount 후에도 setState가 호출됨
- 아직 헷갈리는 점: Strict Mode에서 cleanup이 두 번 실행되는 이유
- 다음에 확인할 것: cleanup이 없을 때 메모리 누수가 실제로 어떻게 생기는지
```

## 기능 회고 템플릿

```md
## 기능: [기능 이름]

### 문제
[어떤 상황에서 어떤 기능이 필요했는가]

### 선택
[무엇을 선택했는가]

### 이유
[다른 선택지와 비교해 왜 이 방법인가]

### 확인
[실패/빈 상태 등 무엇을 검증했는가]

### 다음 개선
[개선 가능한 지점이 있는가]
```

### 기능 회고 예시

```md
## 기능: 게시글 검색

### 문제
검색어와 카테고리로 목록을 필터링해야 했다.

### 선택
검색어는 controlled input으로 두고, 필터 결과는 state로 저장하지 않고 계산했다.

### 이유
필터 결과는 posts, keyword, category에서 계산 가능하므로 별도 state로 두면
동기화 문제가 생긴다. Effect도 필요 없다.

### 확인
빈 결과 UI와 초기화 버튼을 확인했다.
검색어가 있는 상태에서 카테고리를 바꿔도 결과가 올바른지 확인했다.

### 다음 개선
검색어를 URL search params로 연결해 새로고침해도 유지되게 할 수 있다.
```

이 형식은 면접 답변 구조와 거의 같습니다.

## 포트폴리오에 남길 만한 것

API 이름 나열이 아니라 판단 과정이 드러나는 항목이 좋습니다.

- state 위치를 조정한 사례 (왜 부모로 올렸는지, 왜 내렸는지)
- Effect를 제거하거나 정리한 사례
- fetching 상태(loading/error/empty/success)를 나눈 사례
- reducer/context를 도입하거나 도입하지 않은 이유
- 성능을 측정하고 개선한 기록 (before/after 수치)
- 테스트 시나리오와 실패 상태 처리
- custom Hook으로 로직을 분리한 이유

## 나쁜 포트폴리오 설명

```md
React로 Todo 앱을 만들었습니다.
useState, useEffect, useMemo를 사용했습니다.
```

API 이름만 나열하면 판단이 보이지 않습니다.

## 좋은 포트폴리오 설명

```md
Todo 앱에서 추가/삭제/완료 토글을 구현했습니다.
목록 데이터는 부모 컴포넌트가 소유하고, item 컴포넌트는 `onToggle` handler만 호출하도록 분리했습니다.
완료 개수는 별도 state로 저장하지 않고 todos에서 계산해 동기화 문제를 피했습니다.
빈 상태와 실패 상태 UI를 별도로 만들어 사용자가 다시 시도할 수 있게 했습니다.
```

이 설명은 state 위치, handler 전달, derived value 판단, UX 고려가 드러납니다.

## 프로젝트 README 작성 기준

좋은 README는 "처음 보는 사람이 30초 안에 실행할 수 있는가"를 기준으로 씁니다.

```md
# 프로젝트 이름

한 문장 설명.

## 실행

\`\`\`bash
pnpm install
pnpm dev
\`\`\`

## 구현한 기능

- 검색어로 목록 필터링 (계산값, Effect 없음)
- loading/error/empty/success 상태 분리
- 검색어 URL search params 연동

## 배운 점 / 판단한 것

- filteredItems를 state로 저장하지 않고 렌더링 중 계산했다. (이유: 동기화 버그 방지)
- useFetch custom Hook을 추출해 컴포넌트에서 요청 로직을 분리했다.
```

## 학습 주제별 체크리스트 기록

각 단계를 마친 뒤 체크리스트를 남기면 빈 곳이 보입니다.

```md
## 중급 단계 체크리스트

- [x] Effect가 필요한 상황과 필요 없는 상황을 구분했는가?
- [x] loading, error, empty, success 상태를 화면으로 나눴는가?
- [x] 반복되는 상태 로직을 custom Hook으로 분리했는가?
- [ ] URL도 UI 상태의 일부로 연결해봤는가?
- [x] ref를 state와 구분해서 사용했는가?
- [ ] portal로 modal을 구현해봤는가?
```

빈 항목은 다음 학습 방향이 됩니다.

## 읽으면서 생각할 질문

- 오늘 배운 개념을 코드 경험과 연결해 적었는가?
- API 이름보다 선택 이유가 드러나는가?
- 실패하거나 헷갈린 지점도 기록했는가?
- 다음에 개선할 점이 구체적인가?
- 이 기록을 면접 답변으로 바꿀 수 있는가?
- README만 보고 처음 보는 사람이 실행할 수 있는가?
- 체크리스트에서 빠진 항목이 다음 학습 목표가 되는가?
