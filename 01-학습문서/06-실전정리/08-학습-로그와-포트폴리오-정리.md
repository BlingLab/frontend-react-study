# 학습 로그와 포트폴리오 정리

React를 공부했다는 사실보다 중요한 것은 무엇을 만들었고, 어떤 판단을 했고, 무엇을 다시 설명할 수 있는가입니다. 학습 로그는 이 과정을 남기는 문서입니다.

## 학습 로그 템플릿

```md
## 날짜

- 학습 주제:
- 읽은 문서:
- 고친 예제:
- 새로 이해한 개념:
- 아직 헷갈리는 점:
- 다음에 확인할 것:
```

짧아도 꾸준히 남기면 면접 답변과 포트폴리오 설명의 재료가 됩니다.

## 기능 회고 템플릿

```md
## 기능: 게시글 검색

### 문제
검색어와 카테고리로 목록을 필터링해야 했다.

### 선택
검색어는 controlled input으로 두고, 필터 결과는 state로 저장하지 않고 계산했다.

### 이유
필터 결과는 posts, keyword, category에서 계산 가능하므로 별도 state로 두면 동기화 문제가 생긴다.

### 확인
빈 결과 UI와 초기화 버튼을 확인했다.

### 다음 개선
검색어를 URL search params로 연결할 수 있다.
```

이 형식은 면접 답변 구조와 거의 같습니다.

## 포트폴리오에 남길 만한 것

- 단순 구현보다 판단이 드러나는 기능
- state 위치를 조정한 사례
- Effect를 제거하거나 정리한 사례
- fetching 상태를 나눈 사례
- reducer/context를 도입하거나 도입하지 않은 이유
- 성능을 측정하고 개선한 기록
- 테스트 시나리오와 실패 상태 처리

## 나쁜 포트폴리오 설명

```md
React로 Todo 앱을 만들었습니다.
useState, useEffect, useMemo를 사용했습니다.
```

API 이름만 나열하면 판단이 보이지 않습니다.

## 좋은 포트폴리오 설명

```md
Todo 앱에서 추가/삭제/완료 토글을 구현했습니다.
목록 데이터는 부모 컴포넌트가 소유하고, item 컴포넌트는 `onToggle` handler만 호출하도록 분리했습니다.
완료 개수는 별도 state로 저장하지 않고 todos에서 계산해 동기화 문제를 피했습니다.
```

이 설명은 state 위치, handler 전달, derived value 판단이 드러납니다.

## 읽으면서 생각할 질문

- 오늘 배운 개념을 코드 경험과 연결해 적었는가?
- API 이름보다 선택 이유가 드러나는가?
- 실패하거나 헷갈린 지점도 기록했는가?
- 다음에 개선할 점이 구체적인가?
- 이 기록을 면접 답변으로 바꿀 수 있는가?
