## 이 단계의 목표

- 기능을 작게 쪼개서 완성합니다.
- 테스트를 사용자 행동과 결과 중심으로 생각합니다.
- README만 보고 실행 가능한 상태를 만듭니다.
- React 개념을 코드 경험과 연결해 설명합니다.
- 기능별 정상/빈/실패/로딩 시나리오를 테스트 관점으로 적습니다.
- 버그를 React 문제, JavaScript 문제, 데이터 문제, 스타일 문제로 나눕니다.
- 배포 전 빌드, 접근성, 빈 상태, 에러 상태를 체크합니다.
- PR 설명과 면접 답변에 선택 이유와 tradeoff를 남깁니다.
- label, ARIA, focus 관리로 키보드와 스크린리더 접근성을 확보합니다.
- 기초부터 심화까지 배운 개념을 한 기능 안에서 연결합니다.

## 실전 완료 기준

| 항목 | 완료 기준 |
| --- | --- |
| 기능 | 사용자가 시작부터 끝까지 한 흐름을 완료할 수 있습니다. |
| 상태 | loading, error, empty, success 중 필요한 상태가 빠지지 않았습니다. |
| 접근성 | label, button text, focus, error message가 기본적으로 동작합니다. |
| 키보드 | 마우스 없이 키보드만으로 핵심 기능을 사용할 수 있습니다. |
| 테스트 | 최소 핵심 사용자 행동 1개 이상을 검증할 수 있습니다. |
| 빌드 | `pnpm build`가 통과합니다. |
| 문서 | README만 보고 실행과 확인 방법을 알 수 있습니다. |
| 설명 | 왜 이 구조를 선택했는지 짧게 말할 수 있습니다. |

## 실전 정리 루틴

기능 하나를 끝낼 때 아래 순서로 정리합니다.

1. 완료 조건을 사용자 문장으로 적습니다.
2. 정상, loading, empty, error 상태를 각각 확인합니다.
3. 핵심 사용자 행동을 테스트 시나리오로 바꿉니다.
4. 접근성 label, focus, keyboard 흐름을 봅니다.
5. `pnpm build`와 preview로 build 결과를 확인합니다.
6. PR 설명에 선택 이유와 남은 위험을 남깁니다.

## 단계별 연결 질문

| 이전 단계 | 실전에서 확인할 질문 |
| --- | --- |
| 기초 | 데이터가 어떤 컴포넌트 구조로 바뀌는가? |
| 초급 | 어떤 값이 state이고 어떤 값이 계산값인가? |
| 중급 | Effect가 정말 외부 시스템과 동기화하는가? |
| 고급 | state 종류와 실패 복구 전략이 맞는가? |
| 심화 | 성능 최적화가 측정으로 설명되는가? |

실전 정리는 별도 이론이 아니라 앞 단계의 판단을 실제 기능 하나에 연결하는 단계입니다.
