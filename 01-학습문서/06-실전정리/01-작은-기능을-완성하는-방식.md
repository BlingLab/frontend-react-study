# 작은 기능을 완성하는 방식

실전 코드는 큰 앱을 한 번에 만드는 방식보다 작은 기능을 완성하고 확장하는 방식이 좋습니다.

예를 들어 Todo 기능이라면 순서는 이렇게 잡을 수 있습니다.

1. 목록 보여주기
2. 새 항목 추가하기
3. 완료 토글하기
4. 필터 추가하기
5. 저장 기능 추가하기

각 단계에서 state 위치와 컴포넌트 역할을 다시 봅니다.

## 기능을 얇게 자르는 방식

기능을 작게 자른다는 것은 코드량을 줄인다는 뜻이 아닙니다. 사용자가 확인할 수 있는 단위로 자른다는 뜻입니다.

예를 들어 검색 기능은 이렇게 나눌 수 있습니다.

| 단계 | 사용자가 확인할 수 있는 결과 |
| --- | --- |
| 입력창 표시 | 검색어를 입력할 수 있습니다. |
| local 필터 | 입력한 글자에 따라 목록이 줄어듭니다. |
| 빈 상태 | 결과가 없을 때 메시지가 보입니다. |
| URL 연결 | 새로고침해도 검색어가 유지됩니다. |
| 서버 요청 | 검색어로 API 요청을 보냅니다. |
| 실패 처리 | 요청 실패 시 다시 시도할 수 있습니다. |

각 단계가 끝날 때마다 "지금 사용자가 무엇을 할 수 있는가"를 확인합니다.

## vertical slice로 만들기

실전에서는 layer별로 한 번에 만들기보다 작게라도 전체 흐름을 세로로 완성하는 편이 좋습니다.

```txt
UI 껍데기만 완성
-> state 연결
-> handler 연결
-> empty/error 추가
-> API 연결
-> 테스트/문서 정리
```

처음부터 완벽한 폴더 구조를 잡으려 하지 않습니다. 기능이 실제로 움직인 뒤 책임이 보이면 컴포넌트를 나눕니다.

## 예시: 게시글 검색을 작게 완성하기

게시글 검색 기능을 만든다고 하면 처음부터 API, debounce, URL, pagination, skeleton, 테스트를 모두 넣지 않습니다. 사용자가 확인할 수 있는 순서로 얇게 완성합니다.

| 순서 | 구현 | 완료 확인 |
| --- | --- | --- |
| 1 | 정적 검색 화면 | input, 버튼, 결과 영역이 보입니다. |
| 2 | controlled input | 입력한 값이 React state와 연결됩니다. |
| 3 | local mock 검색 | mock 배열이 검색어로 필터링됩니다. |
| 4 | empty state | 결과가 없을 때 메시지가 보입니다. |
| 5 | loading/error state | mock Promise로 지연과 실패를 확인합니다. |
| 6 | 실제 API 연결 | 서버 응답으로 success/empty/error가 나뉩니다. |
| 7 | URL 연결 | 새로고침해도 검색어와 페이지가 유지됩니다. |
| 8 | 테스트/문서 | 핵심 사용자 흐름과 실행 방법을 남깁니다. |

각 단계마다 build가 통과하고, 사용자가 화면에서 변화를 확인할 수 있어야 합니다.

## 상태 설계부터 작게 하기

기능이 커질수록 boolean state가 늘어나기 쉽습니다.

```tsx
const [isLoading, setIsLoading] = useState(false);
const [error, setError] = useState<string | null>(null);
const [posts, setPosts] = useState<Post[]>([]);
```

처음에는 괜찮아 보이지만 곧 애매한 조합이 생깁니다.

```txt
isLoading = true, error = "실패", posts = [이전 결과]
```

이 상태가 "이전 결과를 유지하며 재요청 중"인지, "실패했는데 loading이 안 꺼진 버그"인지 코드만 보고 알기 어렵습니다. 사용자에게 보이는 상태가 명확하다면 union으로 시작하는 편이 좋습니다.

```tsx
type SearchState =
  | { status: "idle" }
  | { status: "loading"; keyword: string }
  | { status: "success"; keyword: string; posts: Post[] }
  | { status: "empty"; keyword: string }
  | { status: "error"; keyword: string; message: string };
```

작은 기능이어도 상태 조합이 사용자 경험을 바꾸면 타입으로 분리합니다.

## 컴포넌트 분리 순서

처음부터 폴더를 많이 나누지 않습니다. 아래 신호가 보일 때 분리합니다.

| 신호 | 분리 후보 |
| --- | --- |
| JSX가 상태 분기 때문에 길어짐 | `SearchResult`, `EmptyState`, `ErrorState` |
| form 입력 로직이 길어짐 | `SearchForm` |
| 목록 item 구조가 반복됨 | `PostItem`, `PostList` |
| 상태 로직이 UI보다 길어짐 | `usePostSearch` |
| 같은 UI 패턴이 두 곳 이상 나옴 | 공통 컴포넌트 또는 helper |

분리 기준은 파일 길이가 아니라 책임입니다. 이름을 붙였을 때 역할이 자연스러우면 분리할 수 있습니다.

## 완료 조건을 먼저 적기

작업을 시작하기 전에 완료 조건을 적습니다.

```md
## 완료 조건

- 사용자가 새 todo를 추가할 수 있다.
- 빈 제목으로 제출하면 input에 focus가 간다.
- 완료 버튼을 누르면 완료 상태가 바뀐다.
- 완료된 todo 수가 표시된다.
- build가 통과한다.
```

완료 조건은 테스트 시나리오와 면접 답변의 재료가 됩니다.

## 완료 조건을 테스트 문장으로 바꾸기

완료 조건은 그대로 테스트 시나리오가 됩니다.

| 완료 조건 | 테스트 문장 |
| --- | --- |
| 사용자가 새 todo를 추가할 수 있다. | 사용자가 제목을 입력하고 추가하면 목록에 보인다. |
| 빈 제목으로 제출하면 input에 focus가 간다. | 빈 값 제출 후 오류 메시지가 보이고 input이 focus된다. |
| 완료 버튼을 누르면 완료 상태가 바뀐다. | 완료 버튼 클릭 후 완료 표시와 요약 개수가 바뀐다. |
| build가 통과한다. | `pnpm build`가 성공한다. |

이렇게 쓰면 구현, 테스트, PR 설명이 같은 언어를 공유합니다.

## 작업 중 범위가 커질 때

기능을 만들다 보면 "이왕이면"이 계속 붙습니다.

```txt
검색만 만들기
-> 정렬도 필요
-> pagination도 필요
-> URL도 필요
-> 최근 검색어도 필요
-> 즐겨찾기도 필요
```

이때는 새 요구를 바로 넣기보다 아래처럼 분리합니다.

| 요구 | 이번 작업에 넣을지 |
| --- | --- |
| 검색어 입력과 결과 표시 | 핵심 흐름이면 포함 |
| 실패/빈 상태 | 사용자에게 바로 보이므로 포함 |
| URL 공유 | 요구사항에 있거나 새로고침 유지가 중요하면 포함 |
| 최근 검색어 | 별도 기능으로 분리 가능 |
| 즐겨찾기 | 검색 기능과 독립된 vertical slice |

작업 범위는 "사용자가 하나의 행동을 끝낼 수 있는가"를 기준으로 자릅니다.

## PR 크기를 조절하는 기준

작은 기능 완성은 PR 크기와도 연결됩니다.

| 좋은 PR | 위험한 PR |
| --- | --- |
| 사용자 흐름 하나가 끝까지 동작 | 여러 기능이 반쯤 섞임 |
| 변경 이유를 3문장으로 설명 가능 | 변경 이유가 파일별로 흩어짐 |
| 테스트/수동 확인이 명확 | 무엇을 확인해야 할지 불명확 |
| rollback해도 영향 범위가 작음 | 되돌리면 여러 기능이 함께 깨짐 |

코드량이 많아도 하나의 사용자 흐름이면 괜찮습니다. 코드량이 적어도 여러 관심사가 섞이면 리뷰와 배포가 어려워집니다.

## 범위를 줄이는 질문

- 지금 꼭 서버 저장이 필요한가, local state로 먼저 확인할 수 있는가?
- 정렬/필터/검색을 모두 한 번에 넣어야 하는가?
- 접근성 요구 중 지금 바로 필요한 최소는 무엇인가?
- 실패 상태를 실제 API 없이 mock으로 먼저 확인할 수 있는가?
- 같은 기능을 파일을 보지 않고 다시 만들 수 있을 만큼 단순한가?

## 읽으면서 생각할 질문

- 지금 만드는 핵심 사용자 흐름은 무엇인가?
- 첫 번째로 동작해야 하는 가장 작은 기능은 무엇인가?
- 컴포넌트가 역할 기준으로 나뉘어 있는가?
- 예제에서 배운 개념이 코드에 어떻게 나타나는가?
- 기능을 추가해도 기존 흐름을 설명할 수 있는가?
- 완료 조건이 테스트 문장으로 바뀔 수 있는가?
- 이번 PR에서 끝낼 사용자 흐름이 하나로 설명되는가?
