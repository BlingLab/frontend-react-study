# 작은 기능을 완성하는 방식

실전 코드는 큰 앱을 한 번에 만드는 방식보다 작은 기능을 완성하고 확장하는 방식이 좋습니다.

예를 들어 Todo 기능이라면 순서는 이렇게 잡을 수 있습니다.

1. 목록 보여주기
2. 새 항목 추가하기
3. 완료 토글하기
4. 필터 추가하기
5. 저장 기능 추가하기

각 단계에서 state 위치와 컴포넌트 역할을 다시 봅니다.

## 기능을 얇게 자르는 방식

기능을 작게 자른다는 것은 코드량을 줄인다는 뜻이 아닙니다. 사용자가 확인할 수 있는 단위로 자른다는 뜻입니다.

예를 들어 검색 기능은 이렇게 나눌 수 있습니다.

| 단계 | 사용자가 확인할 수 있는 결과 |
| --- | --- |
| 입력창 표시 | 검색어를 입력할 수 있습니다. |
| local 필터 | 입력한 글자에 따라 목록이 줄어듭니다. |
| 빈 상태 | 결과가 없을 때 메시지가 보입니다. |
| URL 연결 | 새로고침해도 검색어가 유지됩니다. |
| 서버 요청 | 검색어로 API 요청을 보냅니다. |
| 실패 처리 | 요청 실패 시 다시 시도할 수 있습니다. |

각 단계가 끝날 때마다 "지금 사용자가 무엇을 할 수 있는가"를 확인합니다.

## vertical slice로 만들기

실전에서는 layer별로 한 번에 만들기보다 작게라도 전체 흐름을 세로로 완성하는 편이 좋습니다.

```txt
UI 껍데기만 완성
-> state 연결
-> handler 연결
-> empty/error 추가
-> API 연결
-> 테스트/문서 정리
```

처음부터 완벽한 폴더 구조를 잡으려 하지 않습니다. 기능이 실제로 움직인 뒤 책임이 보이면 컴포넌트를 나눕니다.

## 완료 조건을 먼저 적기

작업을 시작하기 전에 완료 조건을 적습니다.

```md
## 완료 조건

- 사용자가 새 todo를 추가할 수 있다.
- 빈 제목으로 제출하면 input에 focus가 간다.
- 완료 버튼을 누르면 완료 상태가 바뀐다.
- 완료된 todo 수가 표시된다.
- build가 통과한다.
```

완료 조건은 테스트 시나리오와 면접 답변의 재료가 됩니다.

## 범위를 줄이는 질문

- 지금 꼭 서버 저장이 필요한가, local state로 먼저 확인할 수 있는가?
- 정렬/필터/검색을 모두 한 번에 넣어야 하는가?
- 접근성 요구 중 지금 바로 필요한 최소는 무엇인가?
- 실패 상태를 실제 API 없이 mock으로 먼저 확인할 수 있는가?
- 같은 기능을 파일을 보지 않고 다시 만들 수 있을 만큼 단순한가?

## 읽으면서 생각할 질문

- 지금 만드는 핵심 사용자 흐름은 무엇인가?
- 첫 번째로 동작해야 하는 가장 작은 기능은 무엇인가?
- 컴포넌트가 역할 기준으로 나뉘어 있는가?
- 예제에서 배운 개념이 코드에 어떻게 나타나는가?
- 기능을 추가해도 기존 흐름을 설명할 수 있는가?
