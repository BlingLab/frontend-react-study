# 테스트와 배포 감각

테스트는 구현 세부사항보다 사용자의 행동과 결과에 가깝게 생각합니다.

버튼을 클릭하면 항목이 추가되는지, 실패하면 오류 문구가 보이는지, 빈 목록이면 빈 상태가 보이는지 확인합니다.

배포 전에는 실행 방법이 문서에 남아 있어야 합니다. 나중에 다시 볼 때 `pnpm install`, `pnpm dev`, `pnpm build`만으로 흐름이 떠올라야 합니다.

테스트와 배포는 거창한 마무리가 아니라, 내가 만든 코드를 다시 설명 가능하게 만드는 과정입니다.

현재 저장소에는 테스트 러너와 Testing Library 의존성이 아직 설치되어 있지 않습니다. 이 문서의 테스트 코드는 작성 방향을 보여주는 예시이며, 실제 테스트를 추가할 때 `Vitest`, `@testing-library/react`, `@testing-library/user-event`, `@testing-library/jest-dom`을 함께 설치하고 `package.json`에 테스트 명령을 추가합니다.

## 테스트는 사용자 문장으로 시작한다

테스트 시나리오는 내부 함수 이름이 아니라 사용자 행동으로 씁니다.

```md
- 사용자가 할 일을 입력하고 추가 버튼을 누르면 목록에 새 항목이 보인다.
- 사용자가 빈 제목으로 제출하면 오류 메시지가 보인다.
- 사용자가 다시 불러오기를 누르면 loading 상태가 보인다.
```

이 문장이 코드로 바뀌면 Testing Library의 query도 자연스럽게 정해집니다.

```tsx
await user.type(screen.getByLabelText("할 일"), "문서 읽기");
await user.click(screen.getByRole("button", { name: "추가" }));

expect(screen.getByText("문서 읽기")).toBeInTheDocument();
```

## 무엇을 테스트할까

| 기능 | 확인할 것 |
| --- | --- |
| form | 입력, submit, validation message |
| list | item 표시, empty state, key로 인한 이상 동작 |
| async | loading, error, retry, success |
| router | URL 직접 진입, 뒤로 가기, search param |
| modal | 열기, 닫기, focus, escape |
| permission | 버튼 노출 여부, 접근 불가 메시지 |

모든 구현 함수를 테스트할 필요는 없습니다. 사용자가 보는 결과와 중요한 분기부터 테스트합니다.

## 테스트 종류를 구분하기

React 기능을 검증할 때는 테스트 종류를 섞어 쓰되, 목적을 구분합니다.

| 종류 | 확인하는 것 | 예 |
| --- | --- | --- |
| 단위 테스트 | 순수 함수, reducer, formatter | `todoReducer`가 action에 맞게 state를 바꿈 |
| 컴포넌트 테스트 | 사용자 행동과 화면 결과 | form submit 후 목록에 item 표시 |
| 통합 테스트 | 여러 컴포넌트와 라우팅/데이터 흐름 | 검색 페이지에서 URL query와 결과가 함께 바뀜 |
| E2E 테스트 | 실제 브라우저에서 핵심 흐름 | 로그인 후 게시글 작성 완료 |

처음에는 컴포넌트 테스트 하나와 build 확인만으로도 충분합니다. 기능이 커지면 reducer나 custom Hook을 단위 테스트로 분리하고, 핵심 사용자 흐름은 E2E로 남깁니다.

## Testing Library로 생각하는 방식

Testing Library는 사용자가 화면에서 찾는 방식과 가까운 query를 우선합니다.

| 우선순위 | query | 이유 |
| --- | --- | --- |
| 1 | `getByRole` | 버튼, 링크, heading 같은 접근성 역할 기준 |
| 2 | `getByLabelText` | form label과 input 연결 확인 |
| 3 | `getByText` | 사용자에게 보이는 문구 확인 |
| 4 | `getByTestId` | 다른 query로 표현하기 어려운 경우만 |

```tsx
await user.click(screen.getByRole("button", { name: "저장" }));
expect(screen.getByRole("alert")).toHaveTextContent("저장에 실패했습니다.");
```

`className`, 내부 함수 호출 여부, state 변수 이름보다 화면 결과를 봅니다.

## 비동기 UI 테스트

비동기 UI는 loading, success, empty, error를 따로 확인합니다.

```tsx
await user.click(screen.getByRole("button", { name: "검색" }));

expect(screen.getByText("검색 중...")).toBeInTheDocument();
expect(await screen.findByText("React 기초")).toBeInTheDocument();
```

실패 상태는 성공 테스트의 부록이 아닙니다.

```tsx
server.use(
  http.get("/api/posts", () => {
    return HttpResponse.json({ message: "error" }, { status: 500 });
  }),
);

await user.click(screen.getByRole("button", { name: "검색" }));

expect(await screen.findByRole("alert")).toHaveTextContent("검색에 실패했습니다.");
expect(screen.getByRole("button", { name: "다시 시도" })).toBeEnabled();
```

API mocking 도구를 쓴다면 성공/실패/빈 결과를 테스트마다 명확히 세팅합니다. mock이 전역 상태로 새지 않게 테스트 사이에 초기화합니다.

## 무엇을 테스트하지 않을까

테스트가 많아도 잘못된 대상을 보면 유지보수 비용만 늘어납니다.

| 피할 테스트 | 이유 |
| --- | --- |
| `useState`가 호출됐는지 확인 | 구현 세부사항입니다. |
| className 문자열만 확인 | 사용자가 보는 동작과 멀 수 있습니다. |
| 모든 작은 helper를 무조건 테스트 | 중요한 분기 없는 함수는 비용이 더 큽니다. |
| 스냅샷만 대량 추가 | 왜 깨졌는지 알기 어렵습니다. |
| mock 구현을 다시 테스트 | 실제 사용자 흐름과 멀어집니다. |

테스트는 리팩터링을 도와야 합니다. 내부 구조를 조금 바꿨을 뿐인데 테스트가 대량으로 깨진다면 구현 세부사항에 너무 붙어 있을 수 있습니다.

## 배포 전 최소 확인

```bash
pnpm build
```

빌드 통과는 최소 조건입니다. 추가로 아래를 확인합니다.

- 처음 진입 화면이 비어 있지 않은가?
- 예제 선택 URL이 새로고침 후에도 동작하는가?
- 콘솔에 치명적인 오류가 없는가?
- 모바일 폭에서 텍스트가 겹치지 않는가?
- loading/error/empty 상태를 직접 볼 수 있는가?
- README의 실행 명령이 실제로 맞는가?

## TypeScript 오류를 배포 전에 잡기

테스트가 통과해도 TypeScript 오류가 있으면 빌드가 실패합니다.

```bash
pnpm tsc --noEmit
```

`tsc --noEmit`은 파일을 생성하지 않고 타입 오류만 확인합니다. CI 환경에서 build 전에 실행해두면 타입 오류를 일찍 발견합니다.

현재 저장소의 `pnpm build`는 `tsc -b && vite build`를 실행하므로 타입 검사와 번들 빌드를 함께 확인합니다. 별도 CI를 만든다면 `typecheck`, `test`, `build`를 분리해도 좋습니다.

```json
{
  "scripts": {
    "typecheck": "tsc -b --noEmit",
    "test": "vitest",
    "build": "tsc -b && vite build"
  }
}
```

위 예시는 테스트 도구를 추가할 때의 방향입니다. 실제로 추가할 때는 현재 `tsconfig`와 프로젝트 구조에 맞게 명령을 확인합니다.

## 배포 후 smoke test

배포는 빌드 성공으로 끝나지 않습니다. 배포된 결과에서 핵심 흐름을 짧게 확인합니다.

| 화면 | smoke test |
| --- | --- |
| 첫 진입 | 빈 화면 없이 앱 shell이 보입니다. |
| 예제 선택 | 버튼 클릭 시 URL query와 콘텐츠가 바뀝니다. |
| form 예제 | 입력, 제출, validation이 동작합니다. |
| async 예제 | loading/error/empty/success를 볼 수 있습니다. |
| 반응형 | 좁은 화면에서 sidebar와 콘텐츠가 겹치지 않습니다. |

학습 프로젝트에서는 `pnpm preview`로 로컬 build 결과를 확인하는 것만으로도 smoke test 습관을 만들 수 있습니다.

## 테스트 → 빌드 → 배포 순서

작은 프로젝트라도 이 순서를 지키는 습관이 중요합니다.

```md
1. 기능 구현
2. 수동 확인 (loading, error, empty, success)
3. 핵심 시나리오 테스트 작성 또는 확인
4. pnpm tsc --noEmit (타입 오류 확인)
5. pnpm build (빌드 통과)
6. pnpm preview (빌드 결과물 수동 확인)
7. README 업데이트
```

테스트와 빌드 없이 배포하면 "로컬에서는 됐는데"라는 상황이 생깁니다.

## 읽으면서 생각할 질문

- 사용자가 실제로 하는 행동은 무엇인가?
- 테스트가 내부 함수보다 화면 결과를 보고 있는가?
- 실패 상태와 빈 상태를 확인했는가?
- README만 보고 다른 사람이 실행할 수 있는가?
- build가 통과하는 상태로 마무리했는가?
- TypeScript 오류를 build 전에 확인했는가?
- 테스트가 구현 세부사항보다 사용자 결과를 보고 있는가?
- 배포된 결과에서 smoke test를 했는가?
