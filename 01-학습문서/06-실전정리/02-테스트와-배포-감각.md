# 테스트와 배포 감각

테스트는 구현 세부사항보다 사용자의 행동과 결과에 가깝게 생각합니다.

버튼을 클릭하면 항목이 추가되는지, 실패하면 오류 문구가 보이는지, 빈 목록이면 빈 상태가 보이는지 확인합니다.

배포 전에는 실행 방법이 문서에 남아 있어야 합니다. 나중에 다시 볼 때 `pnpm install`, `pnpm dev`, `pnpm build`만으로 흐름이 떠올라야 합니다.

테스트와 배포는 거창한 마무리가 아니라, 내가 만든 코드를 다시 설명 가능하게 만드는 과정입니다.

## 테스트는 사용자 문장으로 시작한다

테스트 시나리오는 내부 함수 이름이 아니라 사용자 행동으로 씁니다.

```md
- 사용자가 할 일을 입력하고 추가 버튼을 누르면 목록에 새 항목이 보인다.
- 사용자가 빈 제목으로 제출하면 오류 메시지가 보인다.
- 사용자가 다시 불러오기를 누르면 loading 상태가 보인다.
```

이 문장이 코드로 바뀌면 Testing Library의 query도 자연스럽게 정해집니다.

```tsx
await user.type(screen.getByLabelText("할 일"), "문서 읽기");
await user.click(screen.getByRole("button", { name: "추가" }));

expect(screen.getByText("문서 읽기")).toBeInTheDocument();
```

## 무엇을 테스트할까

| 기능 | 확인할 것 |
| --- | --- |
| form | 입력, submit, validation message |
| list | item 표시, empty state, key로 인한 이상 동작 |
| async | loading, error, retry, success |
| router | URL 직접 진입, 뒤로 가기, search param |
| modal | 열기, 닫기, focus, escape |
| permission | 버튼 노출 여부, 접근 불가 메시지 |

모든 구현 함수를 테스트할 필요는 없습니다. 사용자가 보는 결과와 중요한 분기부터 테스트합니다.

## 배포 전 최소 확인

```bash
pnpm build
```

빌드 통과는 최소 조건입니다. 추가로 아래를 확인합니다.

- 처음 진입 화면이 비어 있지 않은가?
- 예제 선택 URL이 새로고침 후에도 동작하는가?
- 콘솔에 치명적인 오류가 없는가?
- 모바일 폭에서 텍스트가 겹치지 않는가?
- loading/error/empty 상태를 직접 볼 수 있는가?
- README의 실행 명령이 실제로 맞는가?

## 읽으면서 생각할 질문

- 사용자가 실제로 하는 행동은 무엇인가?
- 테스트가 내부 함수보다 화면 결과를 보고 있는가?
- 실패 상태와 빈 상태를 확인했는가?
- README만 보고 다른 사람이 실행할 수 있는가?
- build가 통과하는 상태로 마무리했는가?
