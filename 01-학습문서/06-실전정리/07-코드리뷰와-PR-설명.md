# 코드리뷰와 PR 설명

코드리뷰와 PR 설명은 "내가 무엇을 바꿨는지"보다 "왜 그렇게 바꿨는지"를 공유하는 과정입니다. React 학습에서도 이 훈련을 하면 면접 답변과 실전 협업이 같이 좋아집니다.

## 좋은 PR 설명 구조

```md
## 변경 내용

- 중급 단계에 ref, derived state, portal 문서 추가
- 고급 단계에 server state와 query cache 문서 추가
- 예제 README에 새 문서 링크 연결

## 선택 이유

- 외부 자료 기준으로 Effect와 server state 분리가 부족했습니다.
- query cache는 fetching 상태를 이해한 뒤 고급에서 다루는 것이 적절했습니다.

## 확인

- pnpm build 통과
- markdown 로컬 링크 검증 통과

## 남은 작업

- TanStack Query 실제 예제 추가 여부 결정
```

## PR 제목 쓰는 법

제목은 "무엇을"이 아니라 "왜/무엇 때문에"에 가까울수록 좋습니다.

| 나쁜 예 | 좋은 예 |
| --- | --- |
| fix bug | fix: 빈 검색어에서 API 요청이 중복 발생하는 버그 수정 |
| add component | feat: 검색 결과가 없을 때 빈 상태 UI 추가 |
| update styles | style: 모바일 폭에서 카드 간격 개선 |
| refactor | refactor: useSearch 커스텀 훅으로 검색 로직 추출 |

prefix 예시: `feat`, `fix`, `refactor`, `test`, `docs`, `style`, `chore`

## 리뷰에서 먼저 볼 것

| 관점 | 질문 |
| --- | --- |
| 기능 | 사용자가 기대한 흐름이 동작하는가? |
| 상태 | source of truth가 중복되지 않았는가? |
| Effect | 외부 시스템 동기화가 맞는가? |
| 타입 | props와 action 타입이 의도를 설명하는가? |
| 접근성 | label, button name, error role이 있는가? |
| 테스트 | 핵심 사용자 행동이 검증되는가? |
| 문서 | 실행/확인 방법이 남아 있는가? |

## 리뷰 코멘트 쓰는 방식

좋은 리뷰 코멘트는 문제, 영향, 제안을 함께 담습니다.

```md
이 값은 `items`와 `keyword`에서 계산할 수 있어서 별도 state로 두면 동기화 문제가 생길 수 있습니다.
렌더링 중 `const filteredItems = ...`로 계산하면 Effect와 추가 렌더링을 줄일 수 있습니다.
```

단순히 "이상합니다"보다 왜 문제가 되는지와 대안을 적습니다.

### 코멘트 강도 표시

팀에 따라 다르지만 아래처럼 표시하면 리뷰이가 처리 우선순위를 잡기 좋습니다.

```md
[blocking] 이 부분은 병합 전에 꼭 바꿔야 합니다. 경쟁 조건이 발생할 수 있습니다.
[suggestion] 이 로직은 custom Hook으로 빼면 재사용하기 좋을 것 같습니다.
[nit] 변수명을 `res`보다 `response`로 쓰면 읽기 쉬울 것 같습니다.
[question] 이 timeout이 1000ms인 이유가 있나요? 궁금해서 여쭤봅니다.
```

- `blocking`: 병합 전 필수 수정
- `suggestion`: 권장이지만 이번 PR 범위 밖이어도 됨
- `nit`: 스타일/가독성, 강제하지 않음
- `question`: 단순 질문, 응답 후 처리 판단

## 내가 받는 리뷰 처리

리뷰를 받으면 바로 고치기 전에 분류합니다.

- 버그인가?
- 스타일/가독성인가?
- 요구사항 해석 차이인가?
- 성능 우려인가?
- 지금 범위에서 처리할 일인가?

모든 리뷰를 즉시 반영해야 하는 것은 아닙니다. 하지만 반영하지 않는다면 이유를 설명할 수 있어야 합니다.

```md
// 반영하지 않는 경우 예시
suggestion 감사합니다. 다음 PR에서 custom Hook으로 분리할 예정입니다.
이번 PR은 기능 완성에 집중하고 있습니다.
```

## 학습 프로젝트에서 셀프 리뷰하기

혼자 작업할 때도 PR을 만들기 전 셀프 리뷰를 합니다.

```md
## 셀프 리뷰 체크리스트

- [ ] 계산 가능한 값을 별도 state로 저장하지 않았는가?
- [ ] Effect cleanup이 있어야 하는 곳에 있는가?
- [ ] 빈 상태, 로딩 상태, 오류 상태를 화면으로 확인했는가?
- [ ] key가 안정적인 식별자를 쓰는가? (인덱스 남용 여부)
- [ ] console.error가 없는가?
- [ ] pnpm build가 통과하는가?
- [ ] 새 파일이 README 또는 index에 연결되었는가?
```

## React 관련 리뷰 포인트 예시

### state 관련

```tsx
// 리뷰 포인트: filtered는 items와 keyword로 계산 가능
const [filtered, setFiltered] = useState(items); // ← 불필요한 state

useEffect(() => {
  setFiltered(items.filter(item => item.name.includes(keyword)));
}, [items, keyword]);

// 개선안
const filtered = items.filter(item => item.name.includes(keyword));
```

### Effect 관련

```tsx
// 리뷰 포인트: cleanup 없어 메모리 누수 가능
useEffect(() => {
  const id = setInterval(tick, 1000);
  // return () => clearInterval(id); ← 없음
}, []);

// 개선안
useEffect(() => {
  const id = setInterval(tick, 1000);
  return () => clearInterval(id);
}, []);
```

### key 관련

```tsx
// 리뷰 포인트: 인덱스 key는 항목 순서가 바뀔 때 state 혼동
{items.map((item, i) => <Card key={i} {...item} />)}

// 개선안
{items.map((item) => <Card key={item.id} {...item} />)}
```

## 읽으면서 생각할 질문

- PR 설명만 읽고 변경 의도를 알 수 있는가?
- 테스트와 빌드 결과가 남아 있는가?
- 리뷰 코멘트가 문제와 영향을 함께 설명하는가?
- 내가 선택한 구조의 tradeoff를 말할 수 있는가?
- 지금 처리할 일과 후속 작업을 구분했는가?
- 코멘트 강도를 명확히 표시해서 리뷰이가 우선순위를 알 수 있는가?
