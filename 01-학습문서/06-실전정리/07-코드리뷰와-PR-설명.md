# 코드리뷰와 PR 설명

코드리뷰와 PR 설명은 "내가 무엇을 바꿨는지"보다 "왜 그렇게 바꿨는지"를 공유하는 과정입니다. React 학습에서도 이 훈련을 하면 면접 답변과 실전 협업이 같이 좋아집니다.

## 좋은 PR 설명 구조

```md
## 변경 내용

- 중급 단계에 ref, derived state, portal 문서 추가
- 고급 단계에 server state와 query cache 문서 추가
- 예제 README에 새 문서 링크 연결

## 선택 이유

- 외부 자료 기준으로 Effect와 server state 분리가 부족했습니다.
- query cache는 fetching 상태를 이해한 뒤 고급에서 다루는 것이 적절했습니다.

## 확인

- pnpm build 통과
- markdown 로컬 링크 검증 통과

## 남은 작업

- TanStack Query 실제 예제 추가 여부 결정
```

## 리뷰에서 먼저 볼 것

| 관점 | 질문 |
| --- | --- |
| 기능 | 사용자가 기대한 흐름이 동작하는가? |
| 상태 | source of truth가 중복되지 않았는가? |
| Effect | 외부 시스템 동기화가 맞는가? |
| 타입 | props와 action 타입이 의도를 설명하는가? |
| 접근성 | label, button name, error role이 있는가? |
| 테스트 | 핵심 사용자 행동이 검증되는가? |
| 문서 | 실행/확인 방법이 남아 있는가? |

## 리뷰 코멘트 쓰는 방식

좋은 리뷰 코멘트는 문제, 영향, 제안을 함께 담습니다.

```md
이 값은 `items`와 `keyword`에서 계산할 수 있어서 별도 state로 두면 동기화 문제가 생길 수 있습니다.
렌더링 중 `const filteredItems = ...`로 계산하면 Effect와 추가 렌더링을 줄일 수 있습니다.
```

단순히 "이상합니다"보다 왜 문제가 되는지와 대안을 적습니다.

## 내가 받는 리뷰 처리

리뷰를 받으면 바로 고치기 전에 분류합니다.

- 버그인가?
- 스타일/가독성인가?
- 요구사항 해석 차이인가?
- 성능 우려인가?
- 지금 범위에서 처리할 일인가?

모든 리뷰를 즉시 반영해야 하는 것은 아닙니다. 하지만 반영하지 않는다면 이유를 설명할 수 있어야 합니다.

## 읽으면서 생각할 질문

- PR 설명만 읽고 변경 의도를 알 수 있는가?
- 테스트와 빌드 결과가 남아 있는가?
- 리뷰 코멘트가 문제와 영향을 함께 설명하는가?
- 내가 선택한 구조의 tradeoff를 말할 수 있는가?
- 지금 처리할 일과 후속 작업을 구분했는가?
