# 테스트 시나리오 작성하기

테스트는 코드를 믿기 위한 문서입니다. 특히 React에서는 "어떤 함수가 호출됐는가"보다 "사용자가 무엇을 했고 화면이 어떻게 바뀌었는가"를 기준으로 작성합니다.

## Given, When, Then

테스트 시나리오는 세 문장으로 시작할 수 있습니다.

```md
Given: 할 일 목록이 비어 있다.
When: 사용자가 제목을 입력하고 추가 버튼을 누른다.
Then: 목록에 새 할 일이 보인다.
```

이 구조는 기능 설명, 테스트 코드, PR 설명에 모두 재사용됩니다.

## 상태별 시나리오

하나의 기능에도 여러 상태가 있습니다.

| 상태 | 테스트 문장 |
| --- | --- |
| 기본 | 목록이 있으면 항목들이 보인다. |
| empty | 목록이 비어 있으면 빈 상태 문구가 보인다. |
| loading | 요청 중이면 버튼이 비활성화되고 loading 문구가 보인다. |
| error | 실패하면 오류 메시지와 다시 시도 버튼이 보인다. |
| success | 성공하면 결과 목록이 보인다. |

성공 케이스만 테스트하면 실제 사용자가 만나는 절반 이상의 상황이 빠질 수 있습니다.

## Testing Library query 우선순위 감각

가능하면 사용자가 접근하는 방식과 가까운 query를 씁니다.

```tsx
screen.getByRole("button", { name: "추가" });
screen.getByLabelText("검색어");
screen.getByText("결과가 없습니다.");
```

`data-testid`는 마지막 선택지로 둡니다. role, label, text로 찾기 어렵다면 UI 접근성 자체가 부족한 신호일 수 있습니다.

## user-event 사용

`user-event`는 실제 사용자 상호작용에 더 가까운 이벤트 묶음을 발생시킵니다.

```tsx
const user = userEvent.setup();

await user.type(screen.getByLabelText("검색어"), "react");
await user.click(screen.getByRole("button", { name: "검색" }));
```

단순히 `onChange` 함수가 호출됐는지가 아니라, 사용자가 입력한 결과로 화면이 바뀌는지를 확인합니다.

## mock을 쓸 때 주의할 점

API 요청을 mock할 때도 사용자 결과를 기준으로 봅니다.

```tsx
expect(await screen.findByText("게시글 제목")).toBeInTheDocument();
```

mock 함수가 호출됐는지만 보면 화면이 실제로 성공 상태를 보여주는지 놓칠 수 있습니다.

## 읽으면서 생각할 질문

- 이 테스트는 사용자 행동으로 읽히는가?
- 성공, 실패, 빈 상태 중 어떤 분기가 빠졌는가?
- 내부 함수 호출보다 화면 결과를 검증하고 있는가?
- query가 접근성 좋은 UI를 유도하고 있는가?
- 테스트 이름만 읽어도 기능 요구사항을 알 수 있는가?
