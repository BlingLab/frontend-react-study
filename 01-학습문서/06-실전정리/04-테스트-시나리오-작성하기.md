# 테스트 시나리오 작성하기

테스트는 코드를 믿기 위한 문서입니다. 특히 React에서는 "어떤 함수가 호출됐는가"보다 "사용자가 무엇을 했고 화면이 어떻게 바뀌었는가"를 기준으로 작성합니다.

현재 저장소의 기본 의존성에는 테스트 도구가 포함되어 있지 않습니다. 아래 코드는 테스트를 추가할 때의 목표 형태이며, 실제로 실행하려면 Vitest와 Testing Library 계열 패키지를 먼저 설치해야 합니다.

```bash
pnpm add -D vitest @testing-library/react @testing-library/user-event @testing-library/jest-dom jsdom
```

## Given, When, Then

테스트 시나리오는 세 문장으로 시작할 수 있습니다.

```md
Given: 할 일 목록이 비어 있다.
When: 사용자가 제목을 입력하고 추가 버튼을 누른다.
Then: 목록에 새 할 일이 보인다.
```

이 구조는 기능 설명, 테스트 코드, PR 설명에 모두 재사용됩니다.

## 상태별 시나리오

하나의 기능에도 여러 상태가 있습니다.

| 상태 | 테스트 문장 |
| --- | --- |
| 기본 | 목록이 있으면 항목들이 보인다. |
| empty | 목록이 비어 있으면 빈 상태 문구가 보인다. |
| loading | 요청 중이면 버튼이 비활성화되고 loading 문구가 보인다. |
| error | 실패하면 오류 메시지와 다시 시도 버튼이 보인다. |
| success | 성공하면 결과 목록이 보인다. |

성공 케이스만 테스트하면 실제 사용자가 만나는 절반 이상의 상황이 빠질 수 있습니다.

## Testing Library query 우선순위 감각

가능하면 사용자가 접근하는 방식과 가까운 query를 씁니다.

```tsx
screen.getByRole("button", { name: "추가" });
screen.getByLabelText("검색어");
screen.getByText("결과가 없습니다.");
```

`data-testid`는 마지막 선택지로 둡니다. role, label, text로 찾기 어렵다면 UI 접근성 자체가 부족한 신호일 수 있습니다.

### query 우선순위 정리

| 우선순위 | query | 언제 |
| --- | --- | --- |
| 1순위 | `getByRole` | button, input, heading, listitem 등 ARIA role |
| 2순위 | `getByLabelText` | label과 연결된 form 요소 |
| 3순위 | `getByPlaceholderText` | placeholder가 의미있을 때 |
| 4순위 | `getByText` | 화면에 보이는 텍스트 |
| 5순위 | `getByDisplayValue` | 현재 입력값 |
| 6순위 | `getByAltText` | 이미지 alt |
| 7순위 | `getByTitle` | title 속성 |
| 마지막 | `getByTestId` | 다른 방법이 없을 때만 |

## user-event 사용

`user-event`는 실제 사용자 상호작용에 더 가까운 이벤트 묶음을 발생시킵니다.

```tsx
const user = userEvent.setup();

await user.type(screen.getByLabelText("검색어"), "react");
await user.click(screen.getByRole("button", { name: "검색" }));
```

단순히 `onChange` 함수가 호출됐는지가 아니라, 사용자가 입력한 결과로 화면이 바뀌는지를 확인합니다.

## 동기 테스트 기본 예시

```tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { TodoApp } from "./TodoApp";

test("할 일을 입력하고 추가하면 목록에 보인다", async () => {
  const user = userEvent.setup();
  render(<TodoApp />);

  await user.type(screen.getByLabelText("할 일"), "문서 읽기");
  await user.click(screen.getByRole("button", { name: "추가" }));

  expect(screen.getByText("문서 읽기")).toBeInTheDocument();
});

test("빈 제목으로 제출하면 오류 메시지가 보인다", async () => {
  const user = userEvent.setup();
  render(<TodoApp />);

  await user.click(screen.getByRole("button", { name: "추가" }));

  expect(screen.getByRole("alert")).toHaveTextContent("제목을 입력해주세요");
});
```

## 비동기 API 테스트

API 요청이 있는 컴포넌트는 fetch를 mock하거나 `vi.fn()`으로 교체합니다. 테스트에서도 loading, error, success 상태를 따로 검증합니다.

```tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { PostSearch } from "./PostSearch";

const mockFetch = vi.fn();
global.fetch = mockFetch;

test("검색하면 결과 목록이 보인다", async () => {
  mockFetch.mockResolvedValueOnce({
    ok: true,
    json: async () => [{ id: "1", title: "React Hooks 정리" }],
  });

  const user = userEvent.setup();
  render(<PostSearch />);

  await user.type(screen.getByLabelText("검색어"), "react");
  await user.click(screen.getByRole("button", { name: "검색" }));

  expect(await screen.findByText("React Hooks 정리")).toBeInTheDocument();
});

test("검색 중에는 loading 문구가 보인다", async () => {
  let resolve: (value: unknown) => void;
  mockFetch.mockReturnValueOnce(
    new Promise((res) => {
      resolve = res;
    })
  );

  const user = userEvent.setup();
  render(<PostSearch />);

  await user.type(screen.getByLabelText("검색어"), "react");
  await user.click(screen.getByRole("button", { name: "검색" }));

  expect(screen.getByText("검색 중...")).toBeInTheDocument();

  resolve!({ ok: true, json: async () => [] });
});

test("검색 실패 시 오류 메시지와 다시 시도 버튼이 보인다", async () => {
  mockFetch.mockRejectedValueOnce(new Error("네트워크 오류"));

  const user = userEvent.setup();
  render(<PostSearch />);

  await user.type(screen.getByLabelText("검색어"), "react");
  await user.click(screen.getByRole("button", { name: "검색" }));

  expect(await screen.findByRole("alert")).toBeInTheDocument();
  expect(screen.getByRole("button", { name: "다시 시도" })).toBeInTheDocument();
});

test("결과가 없으면 빈 상태 문구가 보인다", async () => {
  mockFetch.mockResolvedValueOnce({
    ok: true,
    json: async () => [],
  });

  const user = userEvent.setup();
  render(<PostSearch />);

  await user.type(screen.getByLabelText("검색어"), "없는검색어");
  await user.click(screen.getByRole("button", { name: "검색" }));

  expect(await screen.findByText("결과가 없습니다.")).toBeInTheDocument();
});
```

`findBy*`는 요소가 나타날 때까지 기다립니다. `getBy*`는 지금 당장 있어야 합니다. async 테스트에서 이 차이를 구분하지 않으면 assertion이 실행되기 전에 테스트가 통과되는 것처럼 보일 수 있습니다.

## Custom Hook 테스트 (renderHook)

Custom Hook은 컴포넌트 없이 `renderHook`으로 직접 테스트할 수 있습니다.

```tsx
import { renderHook, act } from "@testing-library/react";
import { useToggle } from "./useToggle";

test("초기값이 false다", () => {
  const { result } = renderHook(() => useToggle());
  expect(result.current.on).toBe(false);
});

test("toggle을 호출하면 값이 반전된다", () => {
  const { result } = renderHook(() => useToggle());

  act(() => {
    result.current.toggle();
  });

  expect(result.current.on).toBe(true);
});

test("초기값을 true로 설정할 수 있다", () => {
  const { result } = renderHook(() => useToggle(true));
  expect(result.current.on).toBe(true);
});
```

`act`는 Hook 안에서 상태 업데이트가 일어날 때 감싸야 합니다. `act` 없이 호출하면 경고가 나옵니다.

### useFetch Hook 테스트

```tsx
import { renderHook, waitFor } from "@testing-library/react";
import { useFetch } from "./useFetch";

const mockFetch = vi.fn();
global.fetch = mockFetch;

test("url이 null이면 idle 상태를 반환한다", () => {
  const { result } = renderHook(() => useFetch(null));
  expect(result.current.status).toBe("idle");
});

test("fetch 성공 시 success 상태가 된다", async () => {
  mockFetch.mockResolvedValueOnce({
    ok: true,
    json: async () => ({ id: 1, name: "Alice" }),
  });

  const { result } = renderHook(() => useFetch("/api/user/1"));

  expect(result.current.status).toBe("loading");

  await waitFor(() => {
    expect(result.current.status).toBe("success");
  });

  if (result.current.status === "success") {
    expect(result.current.data).toEqual({ id: 1, name: "Alice" });
  }
});

test("fetch 실패 시 error 상태가 된다", async () => {
  mockFetch.mockRejectedValueOnce(new Error("서버 오류"));

  const { result } = renderHook(() => useFetch("/api/broken"));

  await waitFor(() => {
    expect(result.current.status).toBe("error");
  });
});
```

## Context 테스트 패턴

Context를 사용하는 컴포넌트는 Provider를 wrapper로 제공합니다.

```tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { ThemeProvider, useTheme } from "./ThemeContext";

function ThemeDisplay() {
  const { theme } = useTheme();
  return <p>현재 테마: {theme}</p>;
}

function ThemeButton() {
  const { setTheme } = useTheme();
  return <button onClick={() => setTheme("dark")}>다크로 변경</button>;
}

function renderWithTheme(ui: React.ReactElement, theme = "light") {
  return render(
    <ThemeProvider initialTheme={theme}>{ui}</ThemeProvider>
  );
}

test("기본 테마가 화면에 보인다", () => {
  renderWithTheme(<ThemeDisplay />);
  expect(screen.getByText("현재 테마: light")).toBeInTheDocument();
});

test("버튼을 누르면 테마가 바뀐다", async () => {
  const user = userEvent.setup();
  renderWithTheme(
    <>
      <ThemeDisplay />
      <ThemeButton />
    </>
  );

  await user.click(screen.getByRole("button", { name: "다크로 변경" }));

  expect(screen.getByText("현재 테마: dark")).toBeInTheDocument();
});
```

wrapper 패턴을 공통화하면 테스트마다 Provider를 반복하지 않습니다.

```tsx
function createWrapper(providers: React.ComponentType<{ children: React.ReactNode }>[]) {
  return function Wrapper({ children }: { children: React.ReactNode }) {
    return providers.reduceRight(
      (acc, Provider) => <Provider>{acc}</Provider>,
      children
    );
  };
}
```

## 타이머와 debounce 테스트

타이머가 있는 코드는 `vi.useFakeTimers()`로 제어합니다.

```tsx
import { renderHook, act } from "@testing-library/react";
import { useDebounce } from "./useDebounce";

describe("useDebounce", () => {
  beforeEach(() => {
    vi.useFakeTimers();
  });

  afterEach(() => {
    vi.useRealTimers();
  });

  test("delay 전에는 초기값을 유지한다", () => {
    const { result, rerender } = renderHook(
      ({ value }) => useDebounce(value, 300),
      { initialProps: { value: "초기값" } }
    );

    rerender({ value: "변경값" });

    expect(result.current).toBe("초기값");
  });

  test("delay 후에 최신 값으로 바뀐다", () => {
    const { result, rerender } = renderHook(
      ({ value }) => useDebounce(value, 300),
      { initialProps: { value: "초기값" } }
    );

    rerender({ value: "변경값" });

    act(() => {
      vi.advanceTimersByTime(300);
    });

    expect(result.current).toBe("변경값");
  });

  test("delay 안에 여러 번 바뀌면 마지막 값만 반영된다", () => {
    const { result, rerender } = renderHook(
      ({ value }) => useDebounce(value, 300),
      { initialProps: { value: "a" } }
    );

    rerender({ value: "b" });
    rerender({ value: "c" });
    rerender({ value: "d" });

    act(() => {
      vi.advanceTimersByTime(300);
    });

    expect(result.current).toBe("d");
  });
});
```

## 테스트 파일 구조

테스트 파일은 기능별로 묶습니다. `describe` 안에서 상태별로 나눌 수 있습니다.

```tsx
describe("PostSearch", () => {
  describe("검색 성공", () => {
    test("결과 목록이 보인다", async () => { ... });
    test("결과가 없으면 빈 상태 문구가 보인다", async () => { ... });
  });

  describe("검색 중", () => {
    test("loading 문구가 보인다", async () => { ... });
    test("검색 버튼이 비활성화된다", async () => { ... });
  });

  describe("검색 실패", () => {
    test("오류 메시지가 보인다", async () => { ... });
    test("다시 시도 버튼이 보인다", async () => { ... });
  });
});
```

## setup 파일 설정

`vitest.config.ts`에서 setup 파일을 등록하면 모든 테스트에서 jest-dom matcher를 쓸 수 있습니다.

```ts
// vitest.config.ts
import { defineConfig } from "vitest/config";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [react()],
  test: {
    environment: "jsdom",
    setupFiles: ["./src/test/setup.ts"],
  },
});
```

```ts
// src/test/setup.ts
import "@testing-library/jest-dom";
```

이 설정이 있으면 `toBeInTheDocument()`, `toHaveTextContent()`, `toBeDisabled()` 같은 matcher를 import 없이 쓸 수 있습니다.

## mock을 쓸 때 주의할 점

API 요청을 mock할 때도 사용자 결과를 기준으로 봅니다.

```tsx
expect(await screen.findByText("게시글 제목")).toBeInTheDocument();
```

mock 함수가 호출됐는지만 보면 화면이 실제로 성공 상태를 보여주는지 놓칠 수 있습니다.

### MSW (Mock Service Worker)

실무에서는 `fetch`를 직접 mock하는 대신 MSW를 쓰면 네트워크 레벨에서 가로채어 더 현실적인 테스트를 할 수 있습니다.

```ts
// src/mocks/handlers.ts
import { http, HttpResponse } from "msw";

export const handlers = [
  http.get("/api/posts", () => {
    return HttpResponse.json([{ id: 1, title: "테스트 게시글" }]);
  }),
];
```

```ts
// src/mocks/server.ts
import { setupServer } from "msw/node";
import { handlers } from "./handlers";

export const server = setupServer(...handlers);
```

```ts
// src/test/setup.ts
import { server } from "../mocks/server";

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());
```

MSW를 쓰면 컴포넌트 코드가 실제 fetch를 그대로 쓰면서도 테스트에서 응답을 제어할 수 있습니다.

## 테스트 작성 체크리스트

```md
- [ ] 테스트 이름이 사용자 행동과 기대 결과로 읽히는가?
- [ ] Given / When / Then 흐름이 코드에 보이는가?
- [ ] 성공, 실패, 빈 상태, loading 중 빠진 케이스가 없는가?
- [ ] 내부 함수 호출보다 화면 결과를 검증하는가?
- [ ] query가 role, label, text 순서를 따르는가?
- [ ] async 테스트에서 findBy와 getBy를 상황에 맞게 쓰는가?
- [ ] 타이머가 있다면 vi.useFakeTimers()를 쓰는가?
- [ ] Context가 필요하다면 wrapper로 제공하는가?
- [ ] Custom Hook은 renderHook으로 직접 테스트하는가?
```

## 읽으면서 생각할 질문

- 이 테스트는 사용자 행동으로 읽히는가?
- 성공, 실패, 빈 상태, loading 중 어떤 분기가 빠졌는가?
- 내부 함수 호출보다 화면 결과를 검증하고 있는가?
- query가 접근성 좋은 UI를 유도하고 있는가?
- `findBy`와 `getBy`를 상황에 맞게 구분해서 썼는가?
- Custom Hook은 컴포넌트 없이 `renderHook`으로 테스트했는가?
- 타이머, debounce가 있는 코드는 `vi.useFakeTimers()`로 제어했는가?
- Context가 필요한 컴포넌트는 wrapper로 Provider를 제공했는가?
